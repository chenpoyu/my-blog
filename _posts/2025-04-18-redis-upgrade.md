## CPU 100% 的警報

週三下午 3 點，我正在寫 code，手機突然響了。

是 Azure 的 alert：**Redis CPU usage > 90% for 5 minutes**。

我打開 Azure Portal 的 Redis 頁面，CPU 使用率已經飆到 100%，而且持續了快 10 分鐘。

「！，怎麼會這樣？」

我們現在還在測試階段，流量根本不大。怎麼會 CPU 就滿了？

## Basic C1 撐不住了

我去看 Redis 的規格：**Basic C1**。

| 規格 | 記憶體 | CPU | 頻寬 | 價格/月 |
|-----|--------|-----|------|--------|
| Basic C1 | 1 GB | 共享 | 低 | $18 |

當初選 Basic C1 是因為：
- 「測試階段，不需要太好的規格」
- 「省錢」
- 「之後再升級」

但現在看來，這個決定有點天真。

Basic tier 的問題：
1. **沒有 SLA**：Azure 不保證可用性
2. **共享 CPU**：跟別人共用運算資源
3. **沒有 persistence**：資料不會存到硬碟，重開就不見
4. **單一節點**：沒有 replica，掛了就掛了

「嗯⋯⋯看來真的該升級了。」

## 問題是：升級要多少錢？

我去看 Standard tier 的價格：

| 規格 | 記憶體 | CPU | 價格/月 |
|-----|--------|-----|--------|
| Standard C1 | 1 GB | 專用 1 core | $73 |
| Standard C2 | 2.5 GB | 專用 2 cores | $146 |
| Standard C3 | 6 GB | 專用 4 cores | $292 |

從 Basic C1 ($18) 升到 Standard C2 ($146)，**一個月多 $128**。

一年就多 $1,536，折合台幣快 5 萬塊。

這不是小錢。

我得去跟客戶說。

## 跟客戶的對話

週四我約了客戶開會。

我直接攤開 Azure Portal 的監控數據：

「你們看，這是 Redis 的 CPU 使用率。現在測試環境只有十幾個人在測，CPU 就已經 100% 了。」

客戶皺眉：「這樣會怎樣？」

「會變慢，甚至 timeout。而且 Basic tier 沒有 SLA，Redis 掛了 Azure 不負責。」

「那要怎麼辦？」

「升級到 Standard C2。但一個月要多 $128 美金。」

客戶沉默了幾秒。

「為什麼當初不直接選好一點的？」

這個問題我早就想過答案：

「因為我們不確定需要多少資源。如果一開始就選 Standard C2，結果發現根本用不到，就浪費錢了。」

「所以現在的策略是：先用小規格測試，確認瓶頸在哪裡，再針對性地升級。這樣比較省錢。」

客戶想了想，點頭：「好，那就升級吧。但你要確保不會再有下一次。」

「了解。」

## 升級過程

Azure 的 Redis 升級很簡單，就是在 Portal 上點幾下：

1. 進入 Redis resource
2. 點 "Scale"
3. 選擇新的 tier 和規格
4. 點 "Save"

但有幾個要注意的：

### 1. 升級會中斷連線

從 Basic 升到 Standard 是**破壞性升級**，會有短暫的 downtime（通常 1-5 分鐘）。

所以我們選在**週五晚上 10 點**升級，那時候測試人員都下班了。

### 2. 資料會保留

雖然會中斷連線，但資料不會遺失。升級完成後，原本的 key 都還在。

### 3. Connection string 不變

好消息是，connection string 不用改。程式碼完全不用動。

## 升級後的效果

週六早上我打開 Azure Portal 檢查，升級已經完成。

看了一下監控數據：

| 指標 | Basic C1 | Standard C2 |
|-----|----------|-------------|
| CPU 使用率 | 90-100% | 15-20% |
| 記憶體使用率 | 60% | 25% |
| 回應時間 (P99) | 50ms | 8ms |

**效能差超多**。

而且 Standard tier 有這些優點：
- **99.9% SLA**：Azure 保證可用性
- **專用 CPU**：不用跟別人搶資源
- **Persistence**：可以開啟 RDB 或 AOF
- **Replica**：有 master-replica 架構，master 掛了會自動切換

我在 Slack 上跟團隊說：「Redis 升級完成，效能好很多。下週可以繼續測試了。」

## 回頭看：該不該一開始就選 Standard？

有人問我：「既然 Standard 這麼好,為什麼不一開始就選？」

我的答案是：

**不確定的時候，先選小的。確定瓶頸後，再升級。**

如果一開始就選 Standard C2：
- 可能發現根本用不到這麼多資源
- 白白浪費 $146/月
- 不知道瓶頸在哪裡（CPU？記憶體？網路？）

現在這樣做：
- 先用 Basic C1 測試，發現 CPU 是瓶頸
- 升級到 Standard C2，問題解決
- 知道問題在哪裡，未來可以更精準地調整

這不是省小錢，是**用數據驗證假設**。

雲端的好處就是：隨時可以調整規格。不用一開始就做完美的決定。

## 下一個問題：SQL Database

Redis 升級完，測試繼續跑。

結果週一又收到 alert：**SQL Database DTU usage > 90%**。

「又來了⋯⋯」

看來 SQL Database 也要升級了。

但這次的問題更複雜，因為 SQL 的規格選擇比 Redis 多太多：DTU 模型、vCore 模型、Serverless、Hyperscale⋯⋯

這又是另一個故事了。
