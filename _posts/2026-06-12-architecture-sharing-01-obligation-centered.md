## 從一場聊天開始

前幾週，幾個以前的同事傳訊息來找我聊。

他們現在做的專案進行到一半，遇到一些狀況。

客戶的想法跟他們的理解有落差，架構上做的幾個判斷也開始出現摩擦，
中間碰到了一些讓整個開發節奏卡住的問題，
想聽看看我有沒有什麼想法。

我沒辦法對外細說他們的業務細節，
但這場聊天讓我想到一個架構設計方向，
很適合這類情境。

我把它整理出來，
因為這個架構不是某種特定產品才用得上的東西。

電信開通、訂閱綁定、服務啟用、多公司平台⋯⋯
只要你的系統是「一個使用者的請求，會展開成多個跨域動作」，
都可以考慮這個方向。

## 這個架構在解什麼問題

很多系統早期都長成這樣：

使用者按下送出，後端接到請求，一路往下跑。
建合約、建訂閱、綁裝置、建帳務、開發票，
全部在同一個大 Service 或同一個 transaction 裡面完成。

系統還小的時候，這樣沒什麼問題。

但系統越長越大之後，問題就慢慢浮出來：

- 任何一步失敗，整件事的狀態就說不清楚
- 多個 domain 的資料混在同一個地方查，邊界越來越糊
- 一個請求裡藏了太多業務邏輯，要追問題、要改需求都很吃力
- 多公司、多資料來源的情況，路由和資料庫切換散落在各處

這個架構要解的，就是這類問題。

## 核心概念：Order 和 Obligation

這個架構最核心的兩個抽象是 **Order** 和 **Obligation**。

**Order** 是使用者看到的那一筆業務請求。

比如說，客戶要申請一個網路加電視的套餐。
對使用者來說，這就是一張訂單。

但對系統來說，這一張訂單不可能靠單一服務、單一資料庫、單一 transaction 完成。
它可能需要：

- 網路合約服務建立合約
- 電視服務建立訂閱
- 設備服務分配裝置
- 帳務服務建立帳戶
- 發票服務開立發票

這每一個，就是一個 **Obligation**。

Obligation 是系統把 Order 拆解之後的責任執行單元。

它不是簡單的任務佇列，也不只是一個名稱。
它有自己的生命週期、狀態、對應的 domain、路由資訊、失敗處理方式，
還有完整的稽核紀錄。

這個差別很重要，後面的篇章會再展開。

## 整個流程的大輪廓

這個架構的主流程大概長這樣：

```
API Gateway / BFF 接到請求
→ Catalog 確認這個 bundle 包含哪些元件
→ Eligibility 確認業務可行性
→ Permission 確認操作是否被允許
→ 建立 Order
→ Obligation Factory 把 Order 拆成多個 Obligation
→ Routing Metadata 決定每個 Obligation 對應哪個資料來源
→ Datasource Resolver 解析出實際的資料庫目標
→ 各個 Owner Service 執行自己的本地 transaction
→ Outbox 記錄事件，Audit Log 記錄決策
→ 失敗時觸發補償或進人工佇列
```

不是一筆大 transaction，
而是拆解、路由、分發、追蹤。

## 為什麼要分這麼細

一開始聽到這個設計，大多數人的第一反應都是：

這樣不會很複雜嗎？

確實比直接呼叫複雜。

但真正複雜的問題，從來不是因為架構太細所以變複雜，
而是因為責任邊界模糊，複雜度被藏起來了。

藏起來的複雜度，
某一天會以 bug、資料不一致、追不到根因的失敗這些形式冒出來。

這個架構把複雜度顯性化：
每個責任單元的狀態可以被追蹤，
失敗有明確的邊界，
補償邏輯有清楚的歸屬。

## 這個系列打算寫幾篇

這篇算是開頭，先把背景和核心概念說清楚。

接下來幾篇會分別聊：

- 請求進來之前的那幾關：Catalog、Eligibility、Permission 各司其職
- Orchestrator 怎麼拆 Obligation，路由和資料來源怎麼解析
- Owner Service 的邊界要守在哪裡，失敗後補償怎麼做

這個架構不是萬用的。

簡單的 CRUD 系統用這套，明顯過度設計。

但如果你的系統牽涉到多個業務 domain、
每個 domain 各自管自己的資料、
一個使用者請求會展開成跨域的多個動作，
這個方向很值得認真考慮。
