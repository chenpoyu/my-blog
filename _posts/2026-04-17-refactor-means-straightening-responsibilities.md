## 重構做到後面，我越來越在意責任邊界

最近在整理後台和 API 的過程裡，
我最常反覆檢查的，不是畫面有沒有對。

而是這幾件事，到底有沒有放在對的位置：

- Controller 該不該知道這麼多
- Service 到底是在協調流程，還是在偷寫查詢
- Repository 是資料存取，還是已經變成半個業務層
- Response / Resource 有沒有真的把輸出格式收乾淨

老系統最常見的問題，不是某一層完全錯。

而是每一層都多做了一點別人的事。

久了之後，你就會得到一個大家都能改，
但誰都不敢保證的系統。

## Controller 只負責接和轉

我希望 Controller 做的事情越單純越好：

- 取參數
- 明確轉型
- 呼叫 Service
- 回傳 Resource / Response

這一層如果開始直接處理大量判斷，
後面通常會變成最難收的地方。

## Service 負責業務協調

Service 應該回答的是：

- 這個功能要串哪些資料
- 哪些條件要先判斷
- 哪些欄位要補齊
- 發生衝突時該怎麼回應

但它不應該直接往下操作資料模型。

一旦 Service 跳過 Repository 直接摸資料，
責任就會開始滑掉。

## Repository 只負責資料存取

這一層要很克制。

很多舊專案最常見的情況，
就是 Repository 越寫越像黑盒子。

裡面混了查詢、判斷、組字串，
甚至連頁面需求都一起塞進去。

短期很方便。  
長期一定很痛。

## Response / Resource 專心處理輸出

這一層被補起來之後，整個 API 會穩很多。

因為輸出欄位、型別、命名規則，都有地方收。

前後端之間的契約，也比較不會越寫越亂。

## AI 很適合抓出責任錯位

這一輪我讓 AI 協助的方式，不是單純生 code。

而是讓它快速指出一些責任錯位的訊號，例如：

- Controller 直接把整包 request 往下丟
- Service 同時處理查詢和格式轉換
- Repository 為了某個頁面偷偷做了過多投影
- 類似輸出格式分散在多個地方重複組裝
- 同一個欄位在不同 API 用不同命名

這些問題很少是單一 bug。

但它們會讓系統每改一次，都比上一次更難控制。

## 我不想再靠記得就好來維持品質

企業系統跑久了，很容易變成這樣：

這個參數其實要先轉一下。  
這個欄位前端習慣用另一個名字。  
這支 API 順便回初始化資料，因為以前這樣比較快。

這些東西一開始可能只是 expedient。

但累積久了之後，就會變成只有少數人記得的潛規則。

而我現在想做的，是把這些潛規則盡量顯性化：

- 輸入怎麼定義
- 分層怎麼切
- 輸出怎麼回
- 搜尋條件要怎麼穿透
- 排序和欄位映射誰負責

把這些講清楚之後，
AI 才不會每次都從錯的地方幫忙。

## 重構不是把 code 變漂亮

我現在越來越覺得，
好的分層不是為了教科書上的整齊。

而是為了讓團隊少做錯誤決策。

當責任拉直之後，很多事情會自然簡單很多：

- 查問題時知道該往哪一層看
- 加功能時知道應該放哪裡
- Review 時知道該用什麼標準看
- 新人接手時不需要先靠口述補世界觀

這次整理下來，我更確定一件事：

**AI 可以幫我加速整理，但真正讓系統變穩的，還是責任有沒有被放回對的位置。**
