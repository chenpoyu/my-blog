## 一年後的變化

距離上次寫 ChatGPT 的心得,剛好過了一年。這一年 AI 發展的速度快到讓人跟不上。

我這一整年的筆記和學習方式，其實已經很大量的仰賴 ChatGPT，實在比以前更方便的許多。

上週 Google 發布了 Gemini,宣稱在多項評測超越 GPT-4。我第一時間就去試用了 Gemini Pro,想看看它跟 ChatGPT 有什麼不同。

但更大的變化其實不是「哪個模型更強」,而是**AI 已經完全融入我的日常工作**。

## Gemini 的第一印象

老實說,Gemini Pro 和 GPT-4 的差異沒有想像中大。都很強,各有優劣。

**Gemini 的優勢**:
- 多模態能力強 (可以理解圖片、程式碼截圖)
- 回應速度快
- 整合 Google 生態系 (可以直接搜尋最新資訊)

**ChatGPT 的優勢**:
- 對話感覺更自然
- Code Interpreter 很好用
- Plugins 生態比較成熟

但對我來說,**重點不是選邊站**。多個都用,視情況選擇:
- 需要最新資訊 → Gemini
- 需要深度推理 → GPT-4 或 Claude Sonnet 4.5
- 開發輔助 → GitHub Copilot (我主要用 Claude Sonnet 4.5 模型)
- 簡單查詢 → 哪個快用哪個

## AI 如何改變我的工作流程

這一年最大的變化是:**AI 從「偶爾用用的工具」變成「每天都在用的夥伴」**。

### 1. 寫技術文件

以前寫 API 文件、架構說明很痛苦。現在我會:

1. 先用 AI 生成初稿 (給它 code + 需求)
2. 自己補充商業邏輯和特殊情境
3. 再讓 AI 潤飾文字,確保清晰易讀

原本要花 2 小時的文件,現在 40 分鐘搞定。

### 2. Code Review

看到有問題的 PR,我會先問 AI:

```
這段代碼有什麼潛在問題?
考慮效能、安全性、可維護性三個面向
```

它常能找到我沒注意到的細節。例如:
- 這個 LINQ 查詢會產生 N+1 問題
- 這個 async 方法沒有 ConfigureAwait(false)
- 這個字串拼接應該用 StringBuilder

當然最後還是要人工判斷,但 AI 能幫我提高 Review 的覆蓋率。

### 3. 學習新技術

最近在研究 .NET 8 的新功能。我的學習流程變成:

1. 問 AI:「.NET 8 有哪些重要新功能?」→ 快速概覽
2. 選一個有興趣的功能,問詳細說明 → 理解原理
3. 讓 AI 生成範例代碼 → 實際動手試
4. 遇到問題再問 AI → 解決疑惑
5. 最後去官方文件確認細節 → 確保正確

比以前「看完整份文件再動手」快很多,而且更有針對性。

### 4. 快速原型

PM 丟來一個新需求:「我們需要一個簡單的報表 API,統計每天的充電站使用量」。

以前我會花半天寫:
- Controller
- Service
- Repository
- DTO
- 單元測試

現在我直接問 AI:

```
用 .NET 8 Minimal API 寫一個報表端點:
GET /api/reports/daily-usage?from=2023-12-01&to=2023-12-31

回傳 JSON:
{ 
  "date": "2023-12-01",
  "totalSessions": 150,
  "totalEnergy": 2500.5
}

使用 Dapper 查詢 SQL Server
```

它生成的代碼 80% 可以直接用,剩下 20% 調整商業邏輯。**30 分鐘就能給 PM 一個可以 demo 的 prototype**。

## 團隊的轉變

一年前提 AI 工具,還有人覺得「這是作弊」。現在整個團隊都在用:

- **Junior 開發者**: 用 AI 學習,寫測試案例
- **Senior 開發者**: 用 AI 做 code review,生成文件
- **我 (技術主管)**: 用 AI 做技術預研,快速驗證方案

上個月的會議,有人提議:「要不要建立團隊的 AI 使用規範?」

我們討論後制定了這些原則:

**✅ 鼓勵使用**:
- 學習新技術
- 生成測試案例、文件
- 快速原型、概念驗證
- 代碼審查輔助

**⚠️ 謹慎使用**:
- 核心商業邏輯 (需要深度理解需求)
- 安全相關代碼 (密碼、加密、權限)
- 效能敏感的代碼 (需要實際測試)

**❌ 禁止**:
- 把公司敏感代碼貼到公開 AI 服務
- 無腦複製 AI 代碼不經審查就 merge

## 遇到的問題

AI 雖然好用,但也不是沒有問題:

**問題 1: 依賴性**

有次網路斷了,發現自己竟然不太想寫代碼。因為習慣了「先問 AI」,沒有 AI 反而不知道怎麼開始。

**解法**: 提醒自己「AI 是輔助,不是拐杖」。重要的能力還是要自己保持。

**問題 2: 過度信任**

Junior 開發者直接把 AI 生成的代碼 merge,結果有 SQL Injection 漏洞。他以為「AI 生成的應該沒問題吧」。

**解法**: Code Review 不能省,AI 生成的代碼一樣要審查。

**問題 3: 成本**

團隊 8 個人都用 ChatGPT Plus ($20/月),一個月就 $160。加上 API 調用費用,開始有點可觀。

**解法**: 開始研究 Local LLM (但這是以後的計畫了)。

## 對未來的展望

一年前我說「AI 會改變開發方式」,現在已經正在發生。

我預測接下來會有更大的變化:

**2024 年可能會看到**:
- AI 深度整合到 IDE (不只是補全,而是真正的「pair programming」)
- Local LLM 成熟,隱私和成本問題得到解決
- 出現專門為「AI 協作」設計的開發流程和工具

**開發者需要的能力**:
- ~~寫最快最多的代碼~~ 
- ✅ 清楚表達需求 (Prompt Engineering)
- ✅ 快速審查和驗證代碼
- ✅ 架構設計和系統思維
- ✅ 商業邏輯理解和溝通

## 寫在最後

Gemini 的登場不是重點,重點是**AI 已經從「新奇的玩具」變成「生產力工具」**。

就像我們現在不會特別強調「我會用 Google 搜尋」,未來可能也不會特別強調「我會用 AI」——因為那會是基本技能。

這一年我的部落格文章也開始大量使用 AI 輔助:
- AI 幫我整理思路
- AI 幫我潤飾文字
- AI 幫我生成代碼範例
- 但核心觀點和經驗還是我自己的

這就是我理想中的「人機協作」:AI 負責重複性和格式化的工作,人負責創意和判斷。
