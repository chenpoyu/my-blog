**以下資訊，若涉及個人資訊，會以模糊方式調整，避免被帶入特定人士縮影**

「小李，這個 Session 過期的問題你們怎麼解的？」週會上，國防組的小陳舉手問。

「欸，我們上個月剛踩過這個坑！」民間組的小李立刻接話：「當時是因為 Redis 的設定問題...」

這個場景，正是我想看到的。四十人分成四個專案組，最怕的就是**資訊孤島**。A 組遇到的問題，B 組可能已經解過了；B 組寫的工具，C 組不知道可以直接用。這種重複浪費，必須想辦法避免。

## 一、發現問題的契機

事情的開端是十月底的一次事件。

公路局組的小張在趕一個緊急需求，需要處理大量 Excel 匯入。他花了兩天時間寫了一個解析工具，結果寫完後小趙看到了：「欸？這個我們民間組三個月前就寫過了啊！而且還支援 CSV 和 JSON 轉換，你直接拿去用就好了。」

小張當下傻眼：「什麼？我完全不知道有這東西！」

我也傻眼。兩天的工時，就這樣浪費掉了。

更扯的是，後來盤點發現：
- **三個組都各自寫了 API 測試工具**（功能差不多，但語法不同）
- **兩個組都解過 Spring Security 的相同問題**（一個花了三天，一個花了兩天）
- **五個人各自整理了 Java Stream 的筆記**（內容 80% 重疊）

那個禮拜的主管會議，我直接跟老闆報告：「我們有知識管理的問題。」

## 二、第一次嘗試：建立文件庫

最直覺的想法就是：建一個知識庫，大家把東西都丟進去。

我們用了 Confluence，規劃了幾個分類：
- **技術解決方案**（記錄踩過的坑和解法）
- **工具與腳本**（可重複使用的程式）
- **開發規範**（coding standard）
- **專案經驗**（事後檢討和心得）

第一個月很熱鬧。大家都很興奮，狂寫文件。

小李寫了「Spring Boot 打包最佳實踐」、小陳寫了「國防專案資安檢測重點」、小趙寫了「Git Workflow 踩坑紀錄」。

但到了第二個月，更新頻率就掉下來了。

第三個月，已經沒什麼人在寫。

為什麼？我問了幾個人：

「太忙了，寫文件沒空。」
「我不知道要寫什麼。」
「我看了文件庫，但找不到我要的東西。」

我這才理解：**光有工具不夠，沒有動力和流程，文件庫就會變成墳場。**

## 三、第二次嘗試：週會技術分享

既然被動寫文件沒用，那就用主動分享的方式。

我在每週的大週會（四個組一起開）加入一個環節：**技術分享 15 分鐘**。

規則很簡單：
1. **每週輪流**，一個組負責分享
2. **主題自選**，可以是踩坑經驗、技術研究、工具介紹
3. **時間控制**，15 分鐘講完，5 分鐘 Q&A

第一週是民間組，小李分享「Redis Session 管理」。

講得很好，大家都覺得有收穫。會後小陳來找我：「Poyu，我們上個月也遇到 Session 問題，如果早點知道這個解法，就不用自己摸索兩天了。」

第二週是國防組，小王分享「資安檢測常見錯誤」。

講到一半，公路局組的小劉突然舉手：「等等，你說的這個 SQL Injection，我們專案好像也有類似問題！」會後小劉立刻去檢查，果然找到三個潛在漏洞。

這個效果，遠遠超過我的預期。

## 四、意外的收穫：跨組交流

技術分享最大的價值，不是知識本身，而是**打破了組與組之間的牆**。

以前四個組各自運作，除了我這個主管，其他人很少跨組互動。民間組的人不知道國防組在做什麼、公路局組不知道派遣組遇到什麼挑戰。

但有了週會分享後，大家開始有話題了。

會後常常看到這種場景：

「小李，你剛才講的那個 Redis 設定檔可以給我看一下嗎？」
「小王,你們資安檢測是用什麼工具？我們也想導入。」
「小趙，你們的 CI/CD pipeline 設定可以參考一下嗎？」

甚至還出現了**跨組結對解題**的情況。

有一次公路局組遇到效能瓶頸，小張在週會上提出來。會後民間組的小李主動說：「我之前處理過類似問題，我們找時間一起看看？」兩人花了一個下午，成功把查詢時間從 12 秒降到 0.8 秒。

這種**自發性的協作**，是我推動知識共享時沒想到的副作用，但絕對是好的副作用。

## 五、進階做法：Code Review 跨組

週會分享上軌道後，我想更進一步。

如果只是聽分享，知識還是單向的。能不能讓大家**更深度地參與彼此的工作**？

於是我提出一個實驗：**Code Review 跨組制度**。

規則是：
- 每個組每個月挑**一個重要的 PR**
- 找**另一個組的資深工程師**來 Review
- Review 時間不超過 1 小時
- Review 重點放在架構和邏輯，不糾結 style

第一次實驗，我讓民間組的小李去 Review 國防組的一個模組。

小李看完後給了三個建議：
1. 「這個快取策略可以改用 LRU，會更有效率」
2. 「這段 Exception handling 可以更細緻」
3. 「測試覆蓋率可以再提高，尤其是邊界條件」

小王（國防組）很受用：「外部視角果然不一樣，我們自己檢查不出來這些問題。」

後來公路局組也讓派遣組的人來 Review，發現了一個潛在的 Memory Leak。

我發現這個做法有兩個好處：
1. **提升程式品質**（多一雙眼睛，少一些 bug）
2. **學習他組經驗**（看別人怎麼寫，吸收不同風格）

雖然會佔用一些時間，但投資報酬率很高。

## 六、固定活動：技術讀書會

週會分享是短平快的知識交流，但有些deeper的主題，15 分鐘講不完。

於是我們又加了一個活動：**每月技術讀書會**。

時間是**每個月最後一個週五下午 2-4 點**，地點在會議室，自由參加（但強烈建議）。

第一次讀書會，主題是「Spring Boot 最佳實踐」。

我讓小李主持，他準備了一份簡報，整理了這半年來的開發經驗。大家邊聽邊討論，還有人當場拿出筆電測試某些作法。兩小時過得很快，會後小趙說：「這種深入討論的感覺很好，比週會講 15 分鐘更有收穫。」

後來幾次的主題：
- **十一月：「Java Stream 進階應用」**（小陳主講，講了很多實戰案例）
- **十二月（今天）：「微服務架構的坑」**（小王主講，分享國防專案的經驗）

我發現參與度出乎意料地高。雖然是下午時段，但每次都有 20 人以上出席，佔團隊的一半。

## 七、實際成效：知識流動起來了

推動這些機制三個月後，我明顯感受到變化。

**數據上的變化：**
- Confluence 月活躍度從 10% 提升到 60%
- 跨組協作的工單從 2 件/月增加到 8 件/月
- 重複解決相同問題的情況減少約 70%

**氛圍上的變化：**
- 大家開始習慣「先問問有沒有人做過」
- 遇到問題不再悶頭自己解，會主動跨組求助
- 新人 onboarding 更快（有文件和前輩可問）

有一次小張跟我說：「Poyu，以前我覺得自己只是公路局組的成員，現在感覺是整個大團隊的一份子。」

這句話讓我很感動。

知識共享的本質，不只是技術的傳遞，更是**團隊認同感的建立**。

## 八、遇到的阻力與解決

當然，不是所有人都買單。

有些資深工程師一開始很抗拒：「我很忙，哪有時間寫文件、做分享？」

我的作法是：
1. **不強制，但鼓勵**（做了會在績效上加分）
2. **主管帶頭做**（我自己也排進分享輪值）
3. **降低門檻**（分享不一定要簡報，口頭講也行）

還有人質疑：「分享出去，別人就學走了，我的價值在哪？」

我跟他說：「你的價值不在於藏私，而在於你能持續產出新的東西。如果你只靠現有知識吃飯，遲早會被淘汰。」

這種心態的轉變需要時間，但慢慢地大家都能理解。

## 九、我的反思

推動知識共享這件事，我學到幾個重點：

1. **工具不是重點，流程和文化才是**
   - 光建 Confluence 沒用，要有機制讓人願意用

2. **主動勝過被動**
   - 與其等人寫文件，不如辦分享會強制輸出

3. **跨組交流創造意外價值**
   - 不只是知識流動，還能增進團隊凝聚力

4. **小步快跑，持續優化**
   - 不要一次搞太複雜，先做簡單的，再慢慢加

5. **高層支持很重要**
   - 我把這件事納入績效考核，執行力就起來了

## 十、結語

回頭看這三個月的努力，我覺得很值得。

團隊的價值，不只是個人能力的總和，而是**如何讓知識在人與人之間流動**。

當每個人都願意分享、願意學習，整個團隊的天花板就會不斷提高。

我常跟團隊說：「你今天教了別人一個東西，看起來你沒賺到什麼。但當你需要幫助的時候，會有十個人願意教你十個東西。這個投資報酬率，絕對划算。」

**知識共享，不是零和遊戲，而是正和遊戲。**

大家都贏，才是真的贏。
