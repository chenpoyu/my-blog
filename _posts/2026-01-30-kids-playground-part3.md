## 從遊樂場到學習工具

[上篇]({% post_url 2026-01-16-kids-playground-part1 %})講了架構規劃，[中篇]({% post_url 2026-01-23-kids-playground-part2 %})講了四款趣味遊戲的設計。做到這裡其實已經蠻像樣了，但我總覺得少了什麼。

讓幼兒園的小朋友試玩了幾天之後，我觀察到一個現象：他們會反覆玩顏色配對，因為每次過關都有撒花和音效，很有成就感。但不太主動嘗試其他遊戲。

另外園長也問我：「小朋友到底每天玩多久？學到什麼了？家長會想知道。」

我答不出來。

遊樂場有了，但缺少學習的內容，也缺少讓家長了解學習進度的方式。所以第三階段的目標很明確：

1. 加入「學習」性質的遊戲——數字和英文字母
2. 建立積分和成就系統——讓小孩有動力探索
3. 做一個學習履歷頁面——讓家長跟老師知道小孩在幹嘛

## 數字學習：三關循序漸進

數字學習遊戲分成三關，概念是由淺入深：

**第一關：認識 1~5**

畫面上會隨機出現一堆 emoji（可能是 🍎、🌟、🐟、🦋 等等），然後問你「有幾個 🍎？」。下面有四個選項讓你點。

這個關卡的核心其實不是「數數」，而是「數量感」。兩三歲的小孩不見得會從一數到五，但他們可以感覺到「很多」跟「很少」的差別，再搭配反覆練習，慢慢就會建立數字跟數量的對應。

**第二關：認識 6~10**

跟第一關機制一樣，但數字範圍拉高到 6~10。emoji 數量多了之後，小孩就不能光用直覺判斷，得真的去數。

**第三關：數字排序 1~10**

畫面上 1 到 10 被打亂，你要按照從小到大的順序一個一個點。每點對一個，數字就會排進上方的序列裡。

這關其實蠻有挑戰性的。小孩得認得每個數字的形狀，然後知道順序。比較小的小朋友目前還沒辦法獨立完成第三關，但前兩關已經玩得蠻順的了。

每一關有 5 題，全部答完會跳出過關彈窗，然後自動進到下一關。全部三關打完之後可以重頭開始。這個「闖關」的感覺讓他們很有動力繼續玩下去。

## ABC 英文字母：三種模式

英文字母遊戲的設計更複雜一些，因為 26 個字母的學習不像數字那麼線性。我做了三個模式讓使用者自己選：

### 📖 認識字母

這是瀏覽模式，沒有對錯，就是一張一張翻閱 A 到 Z。每張卡片上有：

- 大小寫字母（例如 Aa）
- 對應的 emoji（🍎）
- 英文單字（Apple）
- 中文翻譯（蘋果）

小孩可以自己按「上一個」、「下一個」翻來翻去。很多時候他們不是在學字母，是在看 emoji 然後跟同學說「這是獅子！這是企鵝！」——但其實這也是一種學習，他們在建立字母跟具體事物之間的連結。

26 個字母的資料全部寫在一個陣列裡：

```javascript
const ALPHABET_DATA = [
  { letter: 'A', word: 'Apple', emoji: '🍎', chinese: '蘋果' },
  { letter: 'B', word: 'Bear', emoji: '🐻', chinese: '小熊' },
  // ...26 個全部列出
]
```

選詞的原則是：常見、好認、有對應的 emoji。像 X 就用了 Xylophone（木琴），雖然這個字小孩不可能會念，但 🎵 的 emoji 至少讓畫面不會空空的。

### 🎯 字母配對

顯示一個 emoji 和英文單字，問你「這個字的開頭是什麼字母？」。四個選項裡選一個。

例如出現 🐱 Cat，你要選 C。8 題一輪。

這個模式的目的是建立「字母 → 單字 → 圖像」的三向連結。小孩看到貓就會想到 Cat，想到 Cat 就會想到 C。反覆幾次之後印象會很深。

### 🔤 字母排序

隨機挑 5 個連續字母（例如 G、H、I、J、K），打亂順序讓你排。

跟數字排序的邏輯一樣：點對了排進去，點錯了會有提示。8 輪一組。

這三個模式放在一個選單頁面上，小孩可以自己選想玩哪個。學習模式沒有分數壓力，配對和排序則有星星評分，兼顧自由探索跟挑戰性。

## 積分系統：useProgress Hook

這可能是整個專案裡最重要的一個 Hook。

`useProgress` 做的事情很單純：每次遊戲結束的時候，記錄你玩了什麼遊戲、拿到幾顆星、花了多少時間，全部存到 `localStorage` 裡面。

```javascript
const STORAGE_KEY = 'kids-playground-progress'

const defaultProgress = {
  totalStars: 0,
  gamesPlayed: 0,
  history: [],
  achievements: [],
  lastPlayed: null,
}
```

每次過關的時候呼叫 `recordGame()`：

```javascript
recordGame('color-match', '顏色配對', 3, '翻了 6 次')
```

它會做這幾件事：

1. 把這次紀錄加到 `history` 陣列的最前面（保留最近 100 筆）
2. 累加 `totalStars` 和 `gamesPlayed`
3. 檢查是否觸發新的成就

成就的設計：

| 成就 | 條件 | emoji |
|------|------|-------|
| 初次冒險 | 完成第 1 個遊戲 | 🎉 |
| 遊戲達人 | 完成第 10 個遊戲 | 🔥 |
| 星星收藏家 | 累積 50 顆星星 | 🌟 |
| 完美通關 | 任何遊戲拿到 3 顆星 | 👑 |
| 探險家 | 嘗試全部 6 款遊戲 | 🗺️ |

成就的觸發條件刻意設得不難，讓小孩玩個幾次就能解鎖。「初次冒險」第一場就拿到，「完美通關」只要有一次表現好就有。這不是 Dark Souls，是幼兒遊戲，門檻要低、獎勵要多。

把 `useProgress` 整合進六款遊戲的過程還蠻機械化的——每個遊戲的 `showWin` 觸發邏輯裡加上 `recordGame()` 的呼叫就好。這種重複性高的修改正好適合用 Vibe Coding 來處理，一次改六個檔案，幾分鐘搞定。

## 學習履歷頁面

有了積分系統之後，自然要有個地方可以看。

學習履歷頁面分成三個區塊：

**總覽統計**：星星總數、遊戲次數、成就數量，三張卡片一字排開，一目了然。

**成就區**：五個成就的卡片，解鎖的會顯示 emoji 和名稱，未解鎖的是灰色的 🔒。這個設計參考了手遊常見的成就牆，小孩看到灰色的鎖會想要去解開它。

**遊戲紀錄**：列出最近 30 筆遊戲紀錄，每筆包含遊戲名稱、日期時間、星星評分、備註。最下面有一個重置按鈕（需要二次確認），萬一想清掉紀錄重來的時候用。

園長看到這個頁面之後很滿意：「這樣我就可以跟家長說，你家小朋友今天玩了哪些、表現怎麼樣。」

比什麼都不知道好太多了。

## 關於頁面

既然是一個完整的小產品，順手加了一個關於頁面。

內容包括：

- 平台介紹跟定位（2~6 歲學齡前兒童互動學習）
- 六大功能特色
- 六款遊戲的詳細說明
- 使用指南（四個步驟：打開網站 → 選遊戲 → 開始玩 → 看成果）
- 給家長的話（純前端不蒐集資料、正向鼓勵設計、建議遊玩時長）

「給家長的話」那段我覺得蠻重要的。現在很多家長對小孩使用數位產品有疑慮，把這些資訊寫清楚可以降低他們的擔心：

- 所有遊戲純前端，不需要網路連線
- 不蒐集任何個人資料
- 學習紀錄只存在瀏覽器的 localStorage 裡
- 建議每次遊玩 15~20 分鐘

## 首頁的最終面貌

加完學習遊戲跟系統功能之後，首頁也跟著改版了。

最大的變化是把遊戲分成兩區：

- **🎯 趣味遊戲**：顏色配對、動物翻翻樂、數字氣球、形狀排排看
- **📖 學習專區**：數字學習、ABC 英文字母

中間加了一個進度總覽列，顯示目前的星星、遊戲次數、成就數量。點了會跳到學習履歷頁面。

底下加了兩顆按鈕：「📚 學習履歷」和「ℹ️ 關於我們」。

分區的好處是讓家長或老師可以引導小孩：「今天先去學習專區玩一下，再去趣味遊戲好不好？」比起全部混在一起，多了一層結構感。

## RWD：小手也要按得到

響應式設計這塊花了不少時間處理。主要的 breakpoint 設在 768px：

大螢幕：遊戲卡片 2 欄、動物翻翻樂 4 欄、寬鬆的 padding  
手機螢幕：遊戲卡片 2 欄但間距縮小、翻翻樂 3 欄、字體縮小、按鈕撐滿

有幾個容易忽略的細節：

1. **WinModal 的按鈕**：手機上改成垂直排列 + 寬度 100%，不然兩顆按鈕擠在一起很容易按錯
2. **學習履歷的統計卡片**：手機上 padding 跟字體都要縮小，不然三張卡片排不下
3. **成就列表**：桌面用 `auto-fit` 兩欄，手機強制一欄
4. `global.scss` 裡的 `html { font-size: 14px }` 在 768px 以下，讓所有 `rem` 單位自動縮小

另外 `touch-action: manipulation` 這個 CSS 屬性很關鍵，可以防止小孩快速連點觸發雙擊縮放。`user-select: none` 也是必要的，不然小孩拖著手指就會選取文字，畫面亂七八糟。

## Vibe Coding 的整體回顧

三篇寫下來，整個專案從零到完成大概花了我一個週末加幾個晚上的零碎時間。如果完全不靠 AI，光是六款遊戲的 JSX + SCSS 大概就要一兩週了吧。

Vibe Coding 幫到最多的地方：

- **SCSS 樣式**：描述你要什麼效果，AI 直接給你一整組樣式，包含 hover、active、動畫
- **資料結構**：像 26 個字母的資料陣列、成就定義，這種重複性高的東西 AI 很擅長
- **遊戲邏輯的初版**：洗牌、出題、判斷對錯這些，AI 寫出來的通常八九不離十
- **批次修改**：六個遊戲都要加上 `useProgress`，這種機械化的修改一次搞定

Vibe Coding 幫不到的地方：

- **遊戲設計**：要做什麼遊戲、規則怎麼定、難度怎麼拿捏，這些是你自己要想的
- **使用者體驗**：按鈕要多大、顏色要多鮮豔、動畫要多快，要靠實測才知道
- **Bug 的根因分析**：氣球重疊的問題 AI 不會主動發現，是小朋友實際玩才踩到的
- **產品決策**：要不要做積分系統、學習履歷放什麼資訊，這些是你在做產品規劃

所以我的結論跟前面一樣：**AI 跑前 80%，你打磨後 20%**。但那 20% 才是決定這個東西好不好用的關鍵。

## 接下來

目前整個遊戲樂園已經開發完成，預計二月底正式上線，到時候會放到網頁上讓小朋友直接用瀏覽器玩。從一月初幫家人做幼兒園官網，到現在做出一整個遊戲樂園，這條線拉起來其實蠻有意思的。

技術上沒有什麼特別高深的東西，就是 React + SCSS 的基本功。但因為用了 Vibe Coding，整個開發速度快了很多，讓我有更多時間花在「這個遊戲到底好不好玩」跟「小朋友看不看得懂」上面。

幼兒園的小朋友現在每天都會問老師：「今天可以玩遊戲樂園嗎？」聽到這句話的時候覺得，這個 side project 做得蠻值得的。
