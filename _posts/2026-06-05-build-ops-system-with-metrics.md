## 系列最後，回到營運系統

前三篇一路從維運的重要性、流程面、工具面談到這裡。

最後一篇想把它收成一個更完整的題目：

怎麼建一個真的能支撐日常工作的營運系統。

走到營運系統這一層，
有一件事要先放在前面。

維運中心不是把所有工具塞進同一個入口就好。

如果只是把 Grafana、Kibana、Alertmanager、Dockge、Vault 都連在一起，
那它比較像入口網站。

入口網站有用。

但還不一定是營運系統。

真正的營運系統，
應該要讓團隊在混亂的時候做得出判斷。

它要回答的不是：

「我可以點去哪裡？」

而是：

「現在發生什麼事？影響誰？誰要處理？下一步是什麼？」

## 我會先從服務清單開始

如果要建一個營運系統，
我不會先從最漂亮的 dashboard 開始。

我會先做 Service Catalog。

每個服務至少要有幾個基本資料：

- 服務名稱
- 分類
- Owner
- Repo
- 健康檢查 URL
- Dashboard
- Log 查詢入口
- Runbook
- 相關 Incident 和部署紀錄

這些東西很無聊。

但沒有它們，
後面的監控和告警都很難落地。

因為一個告警如果不知道對應哪個服務、哪個負責人、哪個 runbook，
那它只是訊號。

還不是行動。

## 第二步，是把事件收進同一條時間線

營運系統最有用的地方，
通常不是單點功能。

而是事件關聯。

我希望它能把幾種資料收在一起：

- deployment
- alert
- error
- incident
- service status

因為排查時，
人最需要的是時間順序。

例如：

下午兩點部署了新版本。  
兩點零三分 Redis timeout 開始增加。  
兩點零五分 API latency 告警。  
兩點十分客服收到第一個回報。  
兩點十五分 Incident 被開立。

如果這些資訊分散在不同工具，
大家就要花很多時間拼起來。

如果它們在 Ops Timeline 裡能對上，
很多判斷會快很多。

## 第三步，是把流程做成狀態

營運系統不能只顯示資料。

它還要推著事情往前走。

所以 Incident 不應該只是一筆文字紀錄。

它應該有狀態：

Open。  
Investigating。  
Mitigated。  
Resolved。

也應該有 severity、服務、可能原因、處理紀錄、負責人。

這些欄位不是為了管理表單。

而是讓團隊在壓力下仍然知道：

- 現在是否已經有人接手
- 是不是已經先緩解
- 有沒有對外同步
- 還差什麼才能關閉
- 事後要補什麼改善

好的系統不是讓人多填資料。

好的系統是讓重要資訊不要在混亂中消失。

## 第四步，才是 dashboard 和自動化

當服務、事件和流程都比較清楚後，
dashboard 才會真正有意義。

這時候 Grafana 不只是看 CPU。

它可以回答：

- 哪些服務現在 at-risk
- 哪些告警持續太久
- 哪些服務 24 小時內錯誤最多
- 哪些部署後最常跟錯誤一起出現
- 哪些服務還沒有 runbook 或 owner

自動化也一樣。

Ops Runner、n8n、Alertmanager webhook，
都可以幫忙把重複動作收起來。

但我不會一開始就讓它自動做太多。

我會先讓流程跑清楚，
再把確定安全、確定常見、確定有邊界的動作自動化。

例如：

- 收到告警後寫入 timeline
- 部署完成後自動上報
- 產生 Incident 初始紀錄
- 查詢容器最後幾百行 log
- 執行白名單內的清理或重啟任務

自動化的價值，
不是讓人消失。

而是把人從重複動作中拉回判斷位置。

## 關鍵指標要看，但不能亂用

講到營運系統，
最後一定會碰到 KPI。

我對 KPI 的看法比較保守。

指標是拿來改善系統，
不是拿來製造壓力。

如果只拿 MTTR 去壓人，
大家可能會傾向快點關掉 Incident。

如果只看告警數量，
大家可能會把告警關少一點。

如果只看部署次數，
大家可能會一直部署小改動，
但不代表交付品質真的比較好。

所以指標要搭配情境看。

## 我會先看這幾類指標

## 穩定性指標

- 可用率
- p95 / p99 response time
- error rate
- endpoint down 次數
- container restart / OOM 次數
- 磁碟、記憶體、CPU 長期趨勢

這些回答的是：

系統本身穩不穩。

## 維運效率指標

- MTTA，多久有人開始處理
- MTTR，多久恢復服務
- Incident open 太久的數量
- 重複 Incident 比例
- 告警噪音比例
- resolved 後又 reopen 的比例

這些回答的是：

團隊處理問題的節奏健不健康。

## 交付風險指標

- deployment success rate
- change failure rate
- rollback 次數
- Quality Gate 通過率
- Trivy 高風險阻擋次數
- 部署後一小時內的錯誤增加量

這些回答的是：

變更進 production 之後，
有沒有帶來額外風險。

## 維護成熟度指標

- 有 owner 的服務比例
- 有 runbook 的服務比例
- 有健康檢查的服務比例
- 結構化 log 覆蓋率
- 未納入 Vault 的機密數量
- 超過保留期的 log index 或磁碟使用趨勢

這些比較不像傳統 KPI。

但我覺得很重要。

因為它們看的是系統能不能被長期照顧。

## 指標要能導向下一步

一個指標如果看了之後，
沒有人知道下一步要做什麼，
那它就只是漂亮數字。

我比較喜歡這種指標設計：

- 看到告警噪音高，就回頭調整告警規則
- 看到 MTTA 高，就檢查通知路由和 owner
- 看到 rollback 變多，就檢查部署驗證和測試覆蓋
- 看到 runbook 覆蓋低，就排每週補齊最關鍵服務
- 看到 error rate 跟部署高度相關，就加強 release 前後監控

指標不是結論。

指標是提醒團隊要往哪裡改善。

## 我想要的營運系統，是一套判斷節奏

做到最後，
營運系統不應該只是工具集合。

它應該讓團隊形成一種節奏：

- 平常知道哪些服務需要被照顧
- 部署時知道風險有沒有進來
- 異常時知道先看哪裡
- 處理中知道誰負責什麼
- 結束後知道要補什麼機制

這套節奏如果做起來，
維運就不只是救火。

它會慢慢變成一種組織能力。

讓系統的狀態能被看見，
讓事件的關聯能被理解，
讓團隊的行動能被追蹤，
讓指標真的回到改善。

這才是我覺得營運系統真正該做的事。
