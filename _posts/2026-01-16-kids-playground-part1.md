## 起因：幫完幼兒園官網之後的想法

月初的時候幫家人用 Vibe Coding 做了一個幼兒園的官網。那次的經驗讓我對 AI 輔助開發的效率有了新的認識——從規劃到上線，速度比以前快了不少。

做官網的過程中，園長跟我聊到一件事：「現在的小朋友很早就會滑平板了，但外面的幼兒 App 不是塞滿廣告就是要付費，品質也參差不齊。如果有乾淨的小遊戲讓小朋友在教室或家裡玩就好了。」

這句話在我腦子裡轉了兩個禮拜。

我想了一下，做幼兒園官網的技術棧跟做互動小遊戲其實差不多，都是純前端的東西。既然上次 Vibe Coding 的經驗不錯，那就再來一次——這次目標是做一個兒童遊戲樂園，簡單、乾淨、打開瀏覽器就能玩，預計二月底放上網頁讓大家使用。

## 什麼是 Vibe Coding？先講一下我怎麼用

Vibe Coding 這個詞最近蠻常看到的，其實概念很直覺：你把想要做的東西描述清楚，讓 AI 幫你生成大部分的程式碼，你負責把關品質、做決定、調細節。

聽起來好像很偷懶，但實際操作下來，我覺得比較像是從「一行一行敲」變成「一塊一塊組」。你還是得知道自己要什麼、架構要怎麼規劃、哪些地方 AI 寫得不對要改，只是中間那些重複性高的苦工大幅減少了。

上次做幼兒園官網的時候我就體會到這點：頁面結構、表單樣式、RWD 這些 AI 都處理得不錯，但配色、動線、文案語氣還是得自己來。這次做遊戲也是類似的分工——目標明確、元件獨立、邏輯不複雜，正好可以讓 AI 幫忙衝刺，自己專注在遊戲設計跟使用體驗上。

## 先想清楚要做什麼

動手之前我先花了一點時間想整體規劃。畢竟是做給小小孩的東西，不能用大人的邏輯去設計。

**目標使用者**：2~6 歲學齡前兒童

**核心原則**：

- 操作要簡單，點點就能玩，不需要打字或拖放
- 畫面要大、顏色要鮮豔，小手在手機平板上也按得到
- 每個遊戲要短，兩三分鐘一局，小孩注意力就這麼長
- 不設懲罰，答錯了頂多提示一下，不會出現紅色大叉叉嚇到他們
- 純前端就好，不需要後端、不需要帳號登入

想好之後，列了幾個遊戲的雛形：

1. **顏色配對** — 翻牌記憶遊戲，找出一樣的顏色
2. **動物翻翻樂** — 跟顏色配對類似，但用動物圖案
3. **數字氣球** — 天空中飄著氣球，按照 1、2、3 的順序去點
4. **形狀排排看** — 認識基本形狀，放到正確的位置

四個遊戲，涵蓋顏色、動物、數字、形狀，對兩三歲的小孩來說算夠了。

## 技術選型：延續上次做官網的經驗

做幼兒園官網的時候就驗證過這套技術棧了，這次直接沿用，不需要多花時間選型：

- **React 19 + Vite 7**：開發體驗好、打包快，上次做官網就是用這組合
- **React Router DOM 7**：遊戲之間跳轉，用路由管理最乾淨
- **SCSS（BEM 命名法）**：樣式好管理、好覆寫，做 RWD 也方便
- **canvas-confetti**：過關的時候撒花用的，小孩最愛這個
- **Web Audio API**：直接用瀏覽器原生的音訊 API 產生音效，不用額外載入音檔

沒用 TypeScript——不是說 TS 不好，而是這個專案規模小，全部都是 JSX 元件，加 TS 反而拖慢我 Vibe Coding 的節奏。等之後要加功能再看情況。

整個 `package.json` 的 dependencies 就五個套件，乾淨到不行。

## 專案架構設計

資料夾結構是一開始就規劃好的，這樣後面不管加遊戲還是加頁面都很好擴充：

```
src/
├── components/     # 共用元件（返回按鈕、勝利彈窗、星星評分）
├── games/          # 每個遊戲一個資料夾，JSX + SCSS 自己管自己
├── hooks/          # 自訂 Hook（音效、進度系統）
├── pages/          # 頁面（首頁、學習履歷、關於）
└── styles/         # 全域樣式（變數、mixin、動畫、global）
```

每個遊戲都是一個獨立的資料夾，裡面就一個 `.jsx` 跟一個 `.scss`，互不影響。這樣的好處是之後要新增或移除遊戲，不會牽一髮動全身。

共用元件也抽出來了：

- `BackButton`：每個遊戲畫面左上角的返回鍵
- `WinModal`：過關的時候跳出來的慶祝彈窗，含星星評分跟重玩按鈕
- `StarScore`：星星顯示元件

這三個東西幾乎每個遊戲都會用到，不抽出來的話重複程式碼會爆量。

## 樣式系統：為小孩設計的色彩學

這部分我花了不少心思。做兒童介面跟做一般網站最大的差別就是**色彩**和**尺寸**。

全域變數檔 `_variables.scss` 裡我定義了一整組明亮飽和的色系：

- 珊瑚紅 `#FF6B6B`、薄荷綠 `#4ECDC4`、陽光黃 `#FFE66D`
- 薰衣草紫 `#A78BFA`、天空藍 `#60A5FA`、活力橙 `#FB923C`

全部都是高飽和度、高明度的顏色，因為小朋友對鮮豔的顏色反應最好。背景則用柔和的漸層，像是天空漸層 `#87CEEB → #E0F7FA → #FFF8E1`，看起來舒服又不搶戲。

圓角也放很大——最小的圓角是 12px，按鈕類的都是 30px 以上。圓潤的形狀對小孩來說比較親切，尖角看起來會有攻擊性。

另外 `global.scss` 裡還做了幾個重要設定：

```scss
body {
  user-select: none;           // 防止小孩亂拖亂選
  touch-action: manipulation;  // 防止雙擊縮放
}
```

小孩玩的時候什麼都會亂按，這兩行可以避免一堆奇怪的操作。

## 音效系統：用 Web Audio API 自己造

本來想找現成的音效檔，但考量到檔案大小跟載入速度，最後決定直接用 Web Audio API 產生。

`useSound` 這個 Hook 裡面就做了五個音效：

- `playCorrect()`：答對的上行三音（C5 → E5 → G5），愉悅感
- `playWrong()`：答錯的低沉方波，輕柔但明確
- `playClick()`：按鈕點擊的短促提示音
- `playWin()`：過關的八音音階，從 C5 一路上行到 C6
- `playPop()`：戳氣球的隨機高頻音

全部都是用 `OscillatorNode` 產生的正弦波或方波，不用載入任何音檔，首次載入零延遲。而且因為 `exponentialRampToValueAtTime` 的關係，每個音都有自然的衰減，聽起來不會突兀。

## 第一個遊戲：顏色配對

先從最簡單的開始。

規則：畫面上有 8 張卡片（4 對顏色），背面朝上。每次翻兩張，如果顏色一樣就配對成功，不一樣就蓋回去。找到全部 4 對就過關。

做出來之後拿給幼兒園的小朋友試玩，他們居然真的會玩，而且玩得很開心。翻到一樣的顏色會興奮地拍手，聽到過關音效更是笑到不行。

每張卡片上除了顏色底色之外，還放了對應的水果 emoji（紅色是 🍎、藍色是 🫐、綠色是 🥒），讓不認識顏色名稱的小孩也能用圖案來記憶。

星星評分的邏輯很寬鬆：翻 6 次以內找完拿 3 顆星，9 次以內 2 顆，再多就 1 顆。反正重點是鼓勵，不是競爭。

## 先到這邊

這篇先講到整體規劃跟第一個遊戲。下一篇會接著講其他三款趣味遊戲的設計，包括那個讓我重做了一次的動物配對，還有氣球重疊的問題怎麼解。

做給小朋友的東西，動力真的不一樣。尤其上次做完官網之後，能繼續用同樣的技術做出他們真的會玩的遊戲，這種感覺蠻好的。
