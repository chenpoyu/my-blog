**以下資訊，若涉及個人資訊，會以模糊方式調整，避免被帶入特定人士縮影**

「Poyu，我的 KPI 是什麼？」

這是我當上主管後，最常被問也最難回答的問題。

如何為工程師設定 KPI？這真的是管理工作中最棘手的挑戰之一。設得太具體，會扭曲行為；設得太模糊，又沒有意義。

## 第一次設 KPI 的災難

剛升任主管時，HR 要求我為團隊設定 KPI。我天真地想：「工程師嘛，就看產出啊。」

於是我設了這些指標：

**小王（資淺工程師）：**
- 完成 50 個 story points
- 修復 20 個 bugs
- 程式碼行數 5000 行

**小李（資深工程師）：**
- 完成 80 個 story points
- Code Review 50 個 PR
- 帶 2 位新人

看起來很具體、可量化，應該很合理吧？

### 三個月後的結果

**小王的狀況：**

他拼命寫程式，為了達到 5000 行，把很多可以簡化的代碼寫得很冗長。

有次 Code Review，我發現他一個簡單的功能寫了 300 行，我問：「這個可以用工具類簡化吧？」

他說：「可以，但這樣我的程式碼行數就不夠了。」

我當場傻眼。

**小李的狀況：**

為了 Review 50 個 PR，他每個都快速掃過，很多問題沒抓出來。

有次我發現一個明顯的 bug 進了主分支，問他：「你沒 Review 到嗎？」

他說：「我看了，但時間不夠，要趕 KPI。」

**更糟的是：**

團隊開始變得很功利，大家都在「為了 KPI 而工作」，而不是「為了把事情做好而工作」。

我意識到：**我設的 KPI 扭曲了團隊的行為。**

## 常見但有問題的 KPI

我研究了一下業界常用的工程師 KPI，發現每個都有問題：

### 1. 程式碼行數

**看起來合理：**寫越多行，產出越多  
**實際問題：**鼓勵寫冗長、複雜的代碼，而不是簡潔優雅的代碼

經典例子：
```java
// 為了衝行數
String result = "";
if (condition1) {
    result = "A";
} else if (condition2) {
    result = "B";
} else if (condition3) {
    result = "C";
}

// 簡潔的寫法（但行數少）
String result = condition1 ? "A" : condition2 ? "B" : "C";
```

### 2. Bug 數量

**看起來合理：**bug 越少，品質越好  
**實際問題：**可能不敢嘗試新功能、不敢重構、隱藏問題

有工程師跟我說：「我知道這個地方有潛在問題，但改了可能會產生新bug，影響我的 KPI，所以我不敢動。」

### 3. 工時

**看起來合理：**工作越久，越努力  
**實際問題：**鼓勵加班而非效率，而且可能灌水

有次我發現某工程師每天都報 10 小時工時，但實際產出很少。後來發現他大部分時間在摸魚，只是人待在公司而已。

### 4. Story Points 完成數

**看起來合理：**完成越多任務，產出越高  
**實際問題：**為了衝數字，會選簡單的任務做，或灌水 story points

團隊開始出現「story point 通膨」，原本估 3 點的任務，開始估 5 點、8 點。

### 5. 提交次數

**看起來合理：**提交越多，越活躍  
**實際問題：**鼓勵頻繁提交無意義的小改動

有人為了衝提交次數，把一個功能拆成 10 個commit，每個commit改一兩行。

## 我重新設計的 KPI

經過慘痛教訓，我重新設計了 KPI 系統。

核心原則：**不要用單一指標，而是多面向評估。**

### 新的評估架構

**1. 任務完成度 (40%)**
- 是否按時交付
- 交付品質如何
- 有沒有超出預期

**評估方式：**不看行數或 story points，而是看「任務是否真正完成」。

完成的定義：
- 功能正確運作
- 測試充分
- 代碼品質良好
- 文件齊全

**2. 程式品質 (30%)**
- Code Review 的評價
- Bug 產生率（但不是絕對數字）
- 代碼的可維護性
- 技術債的處理

**評估方式：**定性評估為主，由 Tech Lead 和我一起評分。

**3. 團隊協作 (20%)**
- Code Review 的品質（不是數量）
- 幫助其他人的頻率
- 知識分享
- 溝通順暢度

**評估方式：**360度回饋，由團隊成員互評。

**4. 學習成長 (10%)**
- 學習了什麼新技術
- 有沒有主動改善流程
- 有沒有嘗試新的解決方案

**評估方式：**自評 + 主管評估。

### 具體案例

**小王（資淺工程師）的季度評估：**

| 面向 | 評分 | 說明 |
|------|------|------|
| 任務完成度 | 85/100 | 完成 7/8 個任務，品質良好 |
| 程式品質 | 75/100 | 偶有疏漏，但進步中 |
| 團隊協作 | 90/100 | 積極幫助同事，主動分享 |
| 學習成長 | 80/100 | 學習了 Docker，應用於專案 |
| **總分** | **82.5** | **良好** |

這樣的評估，比單純看數字更全面，也更能反映真實表現。

## 還是有挑戰

雖然新系統比較合理，但還是有問題：

### 1. 主觀性

多面向評估難免會有主觀成分。

有工程師質疑：「Poyu，為什麼我的團隊協作只有 70 分？」

我解釋：「根據 360 度回饋，有同事反映你有時候溝通比較不耐煩。」

「可是我明明有幫他們啊！」

這種主觀評估很難讓所有人滿意。

### 2. 不同角色難比較

資淺工程師和資深工程師的 KPI 權重應該不同吧？

資淺：任務完成度 50%，學習成長 20%  
資深：任務完成度 40%，團隊協作 30%

但這樣一來，不同角色的分數就不能直接比較了。

### 3. 短期 vs 長期

有些工作短期看不出成效，但長期很重要。

比如重構代碼、寫文件、改善流程，這些在季度 KPI 很難體現。

### 4. 創新和穩定的平衡

如果太重視任務完成度，會導致大家不敢嘗試新東西（怕失敗）。  
如果太重視學習成長，又可能忽略本職工作。

這個平衡很難拿捏。

## 我現在的做法

經過一年多的實踐和調整，我現在的做法是：

**1. KPI 只是參考，不是唯一依據**

績效評估時，KPI 佔 70%，其他 30% 是我的整體觀察。

有些無法量化的貢獻（像是穩定軍心、協調衝突），我會在那 30% 裡考慮。

**2. 定期溝通期待**

每季初，我會跟每個人 1-on-1，說明這一季我期待什麼。

每季末，一起回顧是否達成，哪裡做得好，哪裡可以改善。

**3. 鼓勵成長而非懲罰失敗**

如果工程師嘗試新技術或改善流程，即使失敗了，我也會給正面評價。

因為我要的是一個「敢於嘗試」的團隊，而不是「只求不出錯」的團隊。

**4. 透明化評估標準**

所有的評估標準都公開，讓大家知道會被如何評分。

這樣可以減少「黑箱作業」的感覺。

**5. 持續調整**

每半年檢討一次 KPI 系統，根據回饋調整。

上個月我還在 KPI 裡加了「技術債處理」這一項，因為大家都不想碰technical debt。

## 反思

設定 KPI 真的是一門藝術，而非科學。

**太具體，會扭曲行為；太模糊，又沒有指導意義。**

我現在的心態是：KPI 是工具，不是目的。

目的是：
- 讓團隊知道什麼是重要的
- 幫助大家成長
- 公平地評估貢獻

如果 KPI 系統讓團隊變得功利、短視、不敢嘗試，那就是失敗的。

**好的 KPI，應該能引導正確的行為，而不是扭曲它。**

這條路還在摸索中。或許永遠找不到完美解，但至少要持續改善，讓它越來越接近我們想要的樣子。
