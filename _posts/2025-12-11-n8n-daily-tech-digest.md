## 上週做完英文機器人之後

做完每日英文教學的 bot 之後，有點上癮了。那個「設定好就不用管，每天自動產出東西」的感覺真的很舒服。

然後就想到另一個困擾我很久的事：技術新聞的閱讀。

我平常追的資訊來源不少——幾個技術 blog、Medium 上的一些出版物、Java Magazine、.NET Blog、還有一些雲端跟 DevOps 相關的。但問題是，每天光是打開這些網站掃一輪就要花不少時間，而且大部分文章掃完標題就知道跟自己無關。

能不能讓 AI 幫我先整理好，每天推一份精華摘要就好？

## 架構跟上週差不多

一樣是 n8n + Ollama + Slack 的組合。差別在資料來源從 CSV 變成 RSS。

流程是這樣：

1. n8n 排程每天下午觸發
2. 同時去抓多個 RSS 來源的最新文章
3. 把標題、連結、圖片、內容丟給 Ollama
4. Ollama 扮演「資深技術架構師」的角色，做篩選、摘要、排序
5. 格式化之後推到 Slack

Prompt 裡我設定了幾個重點規則：

- 中文內容的文章排前面（畢竟中文讀起來快，優先看）
- 摘要要包含**具體的技術點**，不能寫那種「本文探討了微服務架構的最新趨勢」這種廢話。要講清楚是什麼 API 改了、哪個框架出了什麼版本、哪個架構遇到什麼痛點
- 最前面有一段 100-150 字的整體趨勢總結，讓我 10 秒就能掌握今天值不值得細看

輸出格式我也定死了，每篇文章都是固定的標籤格式：`BTN:`（標題）、`URL:`（連結）、`IMG:`（圖片）、`SUM:`（摘要）。這樣後續如果要串接其他平台或做成卡片式呈現，解析起來很容易。

## 踩坑：Java Magazine 沒有 RSS

大部分技術網站都有提供 RSS feed，直接在 n8n 裡用 RSS 節點抓就好，非常無腦。

但 Java Magazine 沒有。

它的網站有內容、有文章列表，但就是不提供 RSS。這代表我不能用標準的 RSS 節點，得自己想辦法。

最後的解法是用 n8n 的 HTTP Request 節點直接打它的頁面，拿回來的是 XML 格式的資料。然後再接一個 Code 節點，自己寫 JavaScript 把 XML 解析成跟其他 RSS 來源一樣的 JSON 結構。

說起來一句話，但實際搞的時候花了不少時間。主要是那個 XML 的結構不太規則，有些欄位有時候有、有時候沒有，要做蠻多防呆。

不過搞定之後就跟其他來源一樣，統一丟給 Ollama 處理，後面的流程完全不用改。

## 跑起來的感覺

每天下午 Slack 會推一份快報過來，大概 8-12 篇文章的摘要，最上面有一段總結。

我的閱讀習慣變成：先看總結，知道今天大概有什麼事。有興趣的再往下滑看個別摘要，真的想深入的才點連結進去讀原文。

以前每天花 30-40 分鐘掃新聞，現在大概 5-10 分鐘就搞定。省下來的時間拿去做真正重要的事，怎麼算都划算。

摘要的品質大致可以接受。偶爾會有太精簡到看不出重點的，但大部分時候比我自己掃標題快很多。反正看到摘要不夠清楚的，就點進原文看嘛，不虧。

## 兩週做了兩個 bot

回頭看，這兩個東西的架構其實是同一套：n8n 排程 → 取資料 → Ollama 處理 → Slack 推送。差別只在資料來源和 prompt。

這就是 n8n 好玩的地方——流程模板化之後，要做新的 bot 只要換資料源跟改 prompt 就好。下週可能再來搞點別的。
