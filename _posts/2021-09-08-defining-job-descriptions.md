**以下資訊，若涉及個人資訊，會以模糊方式調整，避免被帶入特定人士縮影**

團隊從 12 人成長到 35 人後，我發現一個嚴重問題：**很多人不清楚自己該做什麼。**

這不是他們不努力，而是職責定義太模糊。

## 引爆點：一次績效面談

上個月跟工程師小王做績效面談，我說：「你這季的表現還可以，但有些地方可以加強。」

小王很困惑：「Poyu，具體是哪些地方？我覺得該做的我都做了啊。」

「比如 Code Review，你做得不夠積極。」

「可是...沒人跟我說要做 Code Review 啊。我以為只有 Senior 才需要做。」

我愣住了。

再問另一位資深工程師小李：「你有幫新人 Review 代碼嗎？」

「有時候有，但我自己專案也很忙，沒時間每個都看。而且這應該是我的職責嗎？」

我才發現：**原本以為「寫程式」就夠清楚了，但實際上根本不夠。**

## 模糊的職責帶來的問題

當職責定義不清，會出現這些狀況：

### 1. 沒人做的灰色地帶

**案例：文件誰來寫？**

工程師以為只要寫程式就好；  
PM以為工程師會寫技術文件；  
結果沒人寫，最後專案交付時文件一片空白。

**案例：新人誰來帶？**

資深工程師以為主管會安排；  
主管以為資深工程師會主動帶；  
結果新人自己摸索，浪費很多時間。

### 2. 期待落差

**工程師的想法：**「我把功能做出來了，就算完成任務了。」

**主管的期待：**「不只要做出來，還要測試充分、代碼品質好、文件齊全。」

雙方期待不同，考核時就會有衝突。

### 3. 升遷不明確

有工程師問我：「我什麼時候能升 Senior？」

我說：「當你能力達到 Senior 的標準。」

「那 Senior 的標準是什麼？」

我想了想，發現我也說不清楚。是工作年資？技術能力？帶人經驗？專案複雜度？

**沒有明確標準，升遷就會很主觀，容易有爭議。**

## 開始定義 JD

我決定為每個角色定義清楚的 Job Description。

參考了一些業界標準，結合我們團隊的實際情況，我定義了四個層級：

### 資淺工程師 (Junior Engineer)

**核心職責：**
- 在指導下完成開發任務
- 參與每日站會，回報進度
- 學習團隊的開發規範和工具

**技術要求：**
- 掌握至少一種程式語言（Java 或 C# 或 Vue.Js）
- 了解基本的資料結構和演算法
- 能使用 Git 進行版本控制

**不需要做：**
- 系統架構設計
- 帶新人
- 技術決策

**考核標準：**
- 任務完成度：70%
- 學習成長：20%
- 團隊協作：10%

### 資深工程師 (Senior Engineer)

**核心職責：**
- 獨立完成中等複雜度的開發任務
- Review 其他人的代碼
- 協助資淺工程師解決技術問題
- 參與技術方案討論

**技術要求：**
- 精通至少一種程式語言和框架
- 了解系統設計原則
- 有實際專案經驗 3 年以上

**需要做：**
- 每週至少 Review 5 個 PR
- 每月至少一次技術分享
- 帶1-2位資淺工程師

**考核標準：**
- 任務完成度：50%
- 代碼品質：20%
- 團隊貢獻：20%
- 技術深度：10%

### 技術主管 (Tech Lead)

**核心職責：**
- 負責專案的技術架構設計
- 管理 3-5 人的小組
- 評估技術方案和風險
- 協調跨組技術問題

**技術要求：**
- 精通多種技術棧
- 有系統架構設計經驗
- 有帶人經驗

**需要做：**
- 參與需求討論，提供技術評估
- Review 關鍵模組的代碼
- 每月至少兩次 1-on-1 with team members
- 撰寫技術文件

**考核標準：**
- 專案交付：40%
- 技術決策：30%
- 團隊管理：20%
- 技術廣度：10%

### 技術經理 (Engineering Manager)

**核心職責：**
- 管理整個技術團隊
- 制定技術發展方向
- 招聘和培養人才
- 跨部門溝通協調

**技術要求：**
- 有豐富的技術背景
- 有管理經驗 2 年以上
- 了解多種技術領域

**需要做：**
- 參與公司戰略規劃
- 每週至少 3 次 1-on-1
- 主持技術評審會議
- 處理團隊內的衝突和問題

**考核標準：**
- 團隊產出：30%
- 團隊穩定性：25%
- 人才培養：25%
- 跨部門協作：20%

## 實施後的改變

定義好 JD 後，我把它發給全team，並開了一場說明會。

### 正面反應

**小王：**「終於知道自己該做什麼了。原來 Code Review 不是 Senior 的專利，我也可以做。」

**小李：**「現在很清楚 Senior 和 Tech Lead 的差異，我知道要往哪個方向努力了。」

**新人小陳：**「一進來就有明確的 JD，知道公司對我的期待，不會迷失。」

### 負面反應

也有人不太習慣：

**老鳥小張：**「以前做不做 Code Review 看心情，現在變成硬性規定，感覺被約束了。」

我跟他說：「JD 不是約束，是明確化。以前你做了沒人知道，現在做了會算進考核，對你有利。」

### 升遷變清楚了

有位資深工程師問我：「我想升 Tech Lead，需要什麼條件？」

我拿出 JD 說：「你看這裡，Tech Lead 需要有架構設計經驗、帶人經驗。你現在缺的是帶人經驗，我可以讓你帶一個小組，如果表現好，半年後可以升。」

他說：「這樣很清楚，我知道該努力什麼了。」

**清楚的界線，才能有明確的期待。**

## JD 不是一成不變的

定義好 JD 後，我每季都會檢視一次，看是否需要調整。

**調整案例1：增加「文件撰寫」**

發現大家都不寫文件，我在所有層級的 JD 都加上：
- Junior：為自己的代碼寫註釋
- Senior：撰寫模組說明文件
- Tech Lead：撰寫系統架構文件

**調整案例2：區分「專家路線」和「管理路線」**

有些 Senior 不想當主管，但技術很強，我增加了「資深專家」(Senior Specialist) 這個職級：
- 不需要管人
- 專注於技術深度
- 薪資等同 Tech Lead

這樣讓不想管人的技術高手也有發展空間。

## 溝通很重要

定義 JD 只是第一步，更重要的是讓大家理解和認同。

**我的做法：**

1. **新人入職** - 第一天就給 JD，說明清楚
2. **定期回顧** - 每季 1-on-1 時，對照 JD 討論表現
3. **晉升評估** - 依據 JD 標準，客觀評估
4. **公開透明** - JD 放在內部 Wiki，所有人都能看

## 也讓團隊知道：升遷不是看年資

以前有人會說：「我來三年了，應該升 Senior 了吧？」

現在我會說：「升遷不是看年資，而是看能力是否符合下一階的 JD。」

拿出 JD 對照：
- 你能獨立完成任務嗎？✓
- 你有 Review 別人的代碼嗎？✗
- 你有帶過新人嗎？✗

「這兩項還不符合，需要再努力。」

**有客觀標準，爭議就會少很多。**

## 反思

定義 JD 這件事，我應該更早做。

當團隊小的時候，大家都知道彼此在做什麼，職責可以模糊。

但當團隊變大，如果沒有明確的定義，就會亂套。

**管理的本質之一，就是建立清晰的結構和規則。**

JD 就是這個結構的基礎。

現在回頭看，定義 JD 帶來的好處：
- 減少職責糾紛
- 明確期待和考核
- 清楚升遷路徑
- 新人更快上手
- 團隊運作更順暢

這個投資，非常值得。
