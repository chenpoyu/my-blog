## 為什麼要換工具?

上週確定要做壓測之後,我本來想用 **k6**。

以前的專案我都用 k6,很熟悉了。

但這次有個問題:團隊裡有人說:

「這次全部都在 Azure 上,我們應該用 Azure Load Testing。」

「可是 k6 不能整合 Azure Load Testing。」

我查了一下,真的。

**Azure Load Testing 只支援 JMeter 和 Locust,不支援 k6。**

「那就用 Locust 吧。」我說。

但團隊裡有人質疑:「Locust 的效能不是比 k6 差嗎?」

「我們來比較一下。」

## Locust 的優點

Locust 是用 Python 寫測試腳本:

```python
from locust import HttpUser, task, between

class MemberUser(HttpUser):
    wait_time = between(1, 3)
    
    @task
    def get_profile(self):
        self.client.get("/api/member/profile")
    
    @task(3)  # 這個 task 執行頻率是上面的 3 倍
    def get_coupons(self):
        self.client.get("/api/coupons/available")
```

語法很直覺,而且:
- **團隊很多人會 Python**
- **有 Web UI**,可以即時看到測試結果
- **分散式執行**,可以用多台機器一起跑
- **社群活躍**,遇到問題比較容易找到解答

「這些優點聽起來都不錯啊。」那個工程師說。

「對,但也有缺點。」

## Locust 的缺點

### 1. 效能瓶頸

Locust 是用 Python 寫的,而 Python 有 GIL (Global Interpreter Lock) 的限制。

這表示:
- **單一 process 無法完全利用多核心 CPU**
- 要達到高併發,需要跑很多 process
- 每個 process 都會消耗記憶體

如果要模擬 2000 個併發使用者,可能要跑 4-5 個 Locust worker process,每個 process 各 400-500 個使用者。

「這樣不是很麻煩嗎?」

「對,而且要開很多機器來跑 Locust master 和 worker。成本也會增加。」

### 2. 資源消耗

我找了一些數據:

| 工具 | 單台機器可模擬的併發數 | 記憶體消耗 |
|------|---------------------|-----------|
| Locust | ~500-1000 | 高 (Python process) |
| k6 | ~5000-10000 | 低 (Go binary) |
| Gatling | ~2000-5000 | 中 (JVM) |

k6 可以在單台機器上模擬更多併發使用者,因為它是用 Go 寫的,編譯成 binary 執行,效能很好。

「所以如果要模擬 2000 個使用者,k6 一台機器就夠了,Locust 可能要 3-4 台?」

「對。」

## k6 的優點

k6 的測試腳本是用 JavaScript:

```javascript
import http from 'k6/http';
import { check, sleep } from 'k6';

export let options = {
    stages: [
        { duration: '1m', target: 100 },   // ramp up to 100 users
        { duration: '3m', target: 100 },   // stay at 100 users
        { duration: '1m', target: 0 },     // ramp down to 0 users
    ],
};

export default function () {
    let res = http.get('https://api.example.com/api/member/profile');
    
    check(res, {
        'status is 200': (r) => r.status === 200,
        'response time < 500ms': (r) => r.timings.duration < 500,
    });
    
    sleep(1);
}
```

優點:
1. **效能好**: 單台機器可以跑 5000+ 併發
2. **輕量級**: 只要下載一個 binary,不用安裝一堆東西
3. **團隊會 JavaScript**: 前端同事也可以寫測試
4. **整合 Grafana**: 可以用 Grafana 看即時監控

「k6 聽起來很不錯啊。」

「對,但問題是:**k6 不能整合 Azure Load Testing**。」

## Azure Load Testing

Azure Load Testing 是 Azure 提供的壓測服務:
- 上傳 JMeter 或 Locust 腳本
- Azure 會自動分配 agent 機器
- 跑完測試後產生報告
- 可以直接整合 Azure Monitor

這樣的好處是:
- **不用自己架測試機器**
- **可以模擬真實的地理位置** (例如從美國、歐洲、亞洲同時打)
- **自動產生圖表和報告**

「但這要錢吧?」

「對,但不貴。以我們的規模,一次測試大概 $5-10 美金。」

「一個月測 10 次也才 $100,比自己架機器便宜多了。」

## 團隊能力才是關鍵

討論到這裡,那個工程師還是覺得 Locust 比較好:

「可是我們團隊 Python 比較強啊,JavaScript 只有前端會。」

這是個好論點。

**選工具不能只看效能和功能,還要看團隊會不會用。**

如果團隊對 Python 很熟,但對 JavaScript 不熟,那用 Locust 反而比較快。

但我看了一下團隊的組成:
- 後端 3 人: 全部會 JavaScript (雖然不是主要語言)
- 前端 2 人: 當然會 JavaScript
- 我自己: JavaScript 和 Python 都會

「其實 JavaScript 和 Python 我們團隊都會,」我說,「語法都不難。」

「你看,」我把兩邊的 code 放在一起比較:

```javascript
// k6
export default function () {
    http.get('https://api.example.com/api/member/profile');
    sleep(1);
}
```

```python
# Locust
class MemberUser(HttpUser):
    @task
    def get_profile(self):
        self.client.get("/api/member/profile")
```

「k6 確實比較簡潔,沒有 class、沒有 decorator。」

「但 Locust 的 class-based 設計,在處理複雜情境時更有彈性。」

「例如登入後才能操作、要維持 session 狀態,Locust 比較好寫。」

他想了想,點頭:「而且最重要的是能整合 Azure,對吧?」

「對。」

## 最後的決定

經過這一輪討論,我們最終選擇 **Locust + Azure Load Testing**:

| 考量因素 | 決策 |
|---------|-----|
| Azure 整合 | **這是關鍵** - Locust 可以,k6 不行 |
| 團隊能力 | Python 和 JavaScript 都會 |
| 維護成本 | Azure Load Testing 不用架機器 |
| 效能 | 單機效能 k6 較好,但 Azure 會自動分配資源 |
| 複雜情境 | Locust 的 class-based 比較有彈性 |

雖然我以前習慣 k6,但這次為了整合 Azure,還是改用 Locust。

**工具只是手段,能解決問題才是重點。**

「其實 Locust 也不錯,」我跟團隊說,「Python 的生態系更豐富,要串接其他服務也方便。」

「而且有狀態的測試情境(例如登入後才能操作),Locust 寫起來比 k6 直覺。」

## 下一步

2. 寫第一個簡單的測試腳本
3. 本地跑一次看結果
