剛開始主導面試時，我以為自己只是評估候選人。但面試越多，我越發現：**面試其實是雙向學習的過程。**

每個候選人都來自不同的公司、不同的專案，帶著不同的經驗和技術背景。透過面試，我等於在窺探各個公司的技術實踐，學到了很多書上或文章裡學不到的東西。

## 學到的實戰經驗：事件驅動架構

有位候選人之前在一家金融科技公司工作，他們用 Kafka 建置了完整的事件驅動架構。

我好奇問他：「為什麼選擇事件驅動？」

他說：「我們的業務需要即時處理大量交易，傳統的同步 API 調用會有效能瓶頸。改用 Kafka 之後，各個服務可以非同步處理事件，系統整體吞吐量提升了好幾倍。」

接著他分享了實作細節：
- 如何設計 Event Schema
- 如何確保訊息不丟失（at-least-once）
- 如何處理重複訊息（idempotent）
- 如何監控和追蹤事件流

這些都是實戰經驗，比我看十篇文章都有用。面試結束後，我立刻去研究 Kafka，也開始思考我們的系統是否適合導入類似架構。

雖然最後沒有錄取他（因為薪資談不攏），但這場面試讓我學到很多。

## 看見更好的 CI/CD 流程

另一位候選人來自一家新創公司，他們的 CI/CD 流程讓我大開眼界。

我們團隊當時的部署流程還很陽春：
1. 開發在本機測試
2. 推到 Git
3. 手動 SSH 到 server
4. 拉代碼、打包、重啟服務

他聽了之後笑著說：「我們公司已經全自動化了。」

然後他描述他們的流程：
- Git push 後自動觸發 Jenkins
- 自動跑單元測試和整合測試
- 測試通過自動打包 Docker image
- 自動部署到測試環境
- 人工驗證後一鍵部署到正式環境
- 部署失敗自動 rollback

聽完我很慚愧，我們的流程根本是石器時代。

面試結束後，我馬上開始規劃我們的 CI/CD 改善計畫。雖然不能一步到位，但至少有了方向。

後來這位候選人加入了我們團隊，現在正在幫忙建置自動化流程。

## 不同的技術棧思維

面試過程中，我接觸到很多不同的技術棧：

**Python 開發者**談到他們用 Django + Celery 處理非同步任務，讓我知道原來非同步不一定要用訊息佇列。

**前端工程師**分享他們用 React + Redux 管理複雜的狀態，讓我更理解前端的演進。

**DevOps 工程師**談到他們用 Kubernetes + Helm 管理容器，讓我看到容器化的未來趨勢。

每個技術棧都有自己的哲學和最佳實踐，了解這些能拓寬我的技術視野。

## 學習別人的思考方式

除了技術，我更看重候選人的思考方式。

有位候選人在回答「如何優化慢查詢」時，他的思路很系統化：

1. **先量化問題** - 用 profiler 找出慢在哪裡
2. **分析原因** - 是索引問題、還是查詢邏輯問題
3. **評估方案** - 列出幾個可能的優化方向
4. **小步實驗** - 先在測試環境驗證效果
5. **監控效果** - 上線後持續追蹤

這種結構化的問題解決思路，對我後來的管理工作很有啟發。

## 了解業界趨勢

面試不同公司來的候選人，也能了解業界在做什麼。

從金融業來的，談風控和資安；  
從電商來的，談高併發和快取；  
從遊戲業來的，談即時通訊和狀態同步；  
從物聯網來的，談邊緣運算和數據傳輸。

每個領域都有自己的挑戰和解決方案。雖然我們不一定會遇到相同問題，但這些知識會成為我的技術積累。

## 反思自己的不足

有時候候選人問的問題，也會讓我反思。

有位候選人問：「你們團隊的技術債處理機制是什麼？」

我當場愣住了，因為我們根本沒有明確的機制。大家都在趕需求，技術債一直累積。

還有人問：「團隊的技術分享文化如何？」「有沒有 code review 的規範？」「如何評估技術方案的選擇？」

這些問題都戳到我們的痛點，也讓我知道該往哪個方向改善。

## 建立人脈網絡

即使沒有錄取，我也會跟一些優秀的候選人保持聯繫。

後來有幾位成為我的技術討論對象，有問題會互相請教。有些甚至介紹其他優秀人才給我。

**面試不只是篩選，更是建立連結。**

## 總結

面試 100 多人後，我深刻體會到：

**面試不是單向的評估，而是雙向的學習。**

- 候選人展示他們的能力
- 我學習他們的經驗和思維
- 彼此交流對技術的理解

每次面試都是一個小型的技術交流會，只要保持開放的心態，就能從中獲益。

所以現在我很期待每次面試，不只是為了找到適合的人，更是為了學習和成長。

**面試 100 人，就等於窺探 100 個不同的技術世界。**
