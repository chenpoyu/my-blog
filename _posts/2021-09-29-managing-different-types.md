**以下資訊，若涉及個人資訊，會以模糊方式調整，避免被帶入特定人士縮影**

「Poyu，為什麼你對小王和小李的管理方式完全不同？」

這是我的 Tech Lead 在一次閒聊時問我的問題。

我想了想：「因為他們是不同類型的人啊。用同一套方式管所有人，根本行不通。」

這一年來，我最大的體悟是：**管理不是套公式，而是因材施教。**

## 一開始的失敗

剛當上主管時，我天真地以為：「只要公平對待每個人，大家就會滿意。」

於是我制定了一套「標準流程」：

- 每週一次 1-on-1
- 每月一次績效回顧
- 遇到問題先自己想，想不出來再問我
- 所有人用同樣的 KPI 標準

看起來很公平，對吧？

### 結果是災難

**小王（資淺工程師）：**

他每次 1-on-1 都說「沒問題」，但實際上問題一堆。因為他「不好意思問」，怕被覺得能力不夠。

結果任務一延再延，品質也不好。

**小李（資深工程師）：**

他覺得每週 1-on-1 浪費時間，他說：「有問題我會主動找你，不用特地約時間。」

但我堅持「規定就是規定」，結果他越來越不耐煩。

**小陳（外向型）：**

他喜歡在團隊會議上討論，覺得這樣有效率。

但我的風格是「先私下溝通，再開會確認」，他覺得效率很低。

**小張（內向型）：**

他不喜歡在大家面前發言，但我常在會議上點名問他意見。

他每次都很緊張，漸漸開始逃避參加會議。

### 反思

**我犯了一個根本錯誤：把「公平」當成「一視同仁」。**

但公平不是「給每個人一樣的東西」，而是「給每個人他需要的東西」。

就像有人需要梯子爬牆，有人需要長凳，有人根本不需要工具——給每個人一樣的梯子，不叫公平，叫僵化。

## 開始分類

我開始觀察團隊成員的不同特質，試著分類。

當然，人不能簡單分類，但理解他們的傾向，可以幫助我調整管理方式。

### 按經驗分類

**1. 資淺工程師（0-2年）**

**特質：**
- 技術基礎還在建立
- 缺乏自信，害怕犯錯
- 需要頻繁的指導和回饋
- 不確定「什麼是好的代碼」

**管理方式：**
- **頻繁的 1-on-1**（每週一次）
- **具體的任務拆解**（不要給模糊的需求）
- **即時的正面回饋**（做對了馬上肯定）
- **錯誤是學習機會**（不要責怪）

**案例：小王**

小王剛來時，我給他一個任務：「實作會員登入功能。」

他做了兩週，結果一團糟——沒有加密、沒有驗證、沒有錯誤處理。

我意識到：不是他能力不夠，而是我沒有給清楚的指引。

後來我改變方式：
```
任務：實作會員登入功能
要求：
1. 使用 BCrypt 加密密碼
2. 加入 JWT Token 機制
3. 處理以下錯誤情境：
   - 帳號不存在
   - 密碼錯誤
   - Token 過期
4. 參考範例：UserService.java
5. 完成後找我 Code Review
```

這次他做得很好。**資淺工程師需要的是清晰的指引，而不是模糊的自由。**

**2. 中階工程師（2-4年）**

**特質：**
- 技術能力已建立
- 開始想要更多自主權
- 在意「為什麼」而不只是「怎麼做」
- 想證明自己可以獨當一面

**管理方式：**
- **給予挑戰性任務**（讓他們證明能力）
- **解釋背景和原因**（不要只下指令）
- **允許失敗空間**（鼓勵嘗試）
- **提供成長機會**（帶 Junior、主導小專案）

**案例：小劉**

小劉技術不錯，但總覺得「沒有發揮空間」。

有次我交代他：「這個功能照舊架構實作就好。」

他回：「為什麼要照舊架構？我覺得用新的方式會更好。」

以前的我會說：「照做就對了，不要問那麼多。」

但我學到：**中階工程師需要的是理解和參與感。**

於是我說：「你覺得哪種方式更好？」

他提出一個方案，我分析給他聽：
- 你的方案確實更優雅
- 但考慮到維護成本，舊架構更適合
- 如果你想試新方案，可以在下一個專案試試

他理解了，而且感覺被尊重了。

**3. 資深工程師（4年以上）**

**特質：**
- 技術能力強
- 不需要手把手指導
- 在意影響力和成就感
- 可能對重複性工作感到厭倦

**管理方式：**
- **給予高度自主權**（信任他們的判斷）
- **討論而非指令**（把他們當夥伴）
- **提供更大的舞台**（跨團隊影響力）
- **尊重專業意見**（即使你不同意）

**案例：小陳**

小陳是團隊最資深的工程師，有次我想導入新的開發流程。

我原本打算直接宣布，但想起他是資深工程師，應該徵詢意見。

1-on-1 時我問他：「我在考慮改用 GitFlow，你覺得呢？」

他說：「GitFlow 對大團隊有用，但我們只有 12 人，太複雜了。我建議用 GitHub Flow 就好。」

他分析了兩種流程的優劣，我發現他說得有道理。

最後我們一起決定用 GitHub Flow，而且由他來主導推行。

**結果：他覺得被尊重，而且全力支持新流程。**

### 按性格分類

除了經驗，性格也很重要。

**1. 外向型 vs 內向型**

**外向型（小明）：**
- 喜歡在會議上討論
- 思考時需要說出來
- 喜歡團隊協作

**管理方式：**
- 開會時多請他發言
- 重要決策在會議上討論
- 安排他跟其他人合作

**內向型（小張）：**
- 需要時間獨自思考
- 不喜歡在大家面前發言
- 書面溝通更自在

**管理方式：**
- 重要議題提前給他看，讓他有時間思考
- 不要在會議上突然點名
- 可以用 Slack 或 Email 討論

**真實案例：**

有次開技術會議，我問：「這個架構大家覺得如何？」

小明馬上說：「我覺得有問題，這裡應該...」

小張沒說話。

以前我會以為「沒說話就是同意」。

但後來我發現，小張只是需要時間思考。

會後我 Slack 他：「對剛剛的架構，你有什麼想法嗎？」

他回了一長串分析，比小明的意見還深入。

**2. 追求完美型 vs 快速交付型**

**追求完美型（小李）：**
- 在意代碼品質
- 重構狂熱者
- 可能進度較慢

**管理方式：**
- 給予充足時間
- 肯定他的品質
- 但也要提醒「Done is better than perfect」

**快速交付型（小王）：**
- 重視速度
- 先做出來再說
- 可能忽略品質

**管理方式：**
- 肯定他的效率
- 但要加強 Code Review
- 教導「快但好」的平衡

**真實案例：**

同樣一個功能，小李做了兩週還在重構，小王三天就交了。

以前我會覺得「小李太慢」、「小王太粗糙」。

但現在我學會：
- 重要且複雜的功能，交給小李
- 緊急且簡單的功能，交給小王
- 讓他們互相 Code Review，互相學習

**3. 自驅型 vs 需要推動型**

**自驅型（小陳）：**
- 主動學習
- 不用催就會做
- 需要的是方向而非管理

**管理方式：**
- 給予目標，不要管太細
- 定期確認進度即可
- 提供資源和支持

**需要推動型（小趙）：**
- 需要明確指令
- 需要定期追蹤
- 不推就不動

**管理方式：**
- 拆解任務，設定 milestone
- 定期追蹤進度
- 給予即時回饋

**真實案例：**

我給小陳和小趙同樣的任務：「研究 Kubernetes 的可行性。」

小陳：一週後給我一份完整報告，包括優缺點分析、學習曲線、成本評估。

小趙：一週後說「我還在看」，兩週後還是「我還在看」。

我學到：**不能用同樣方式對待不同類型的人。**

對小陳：「研究 Kubernetes 的可行性，兩週後給我報告。」  
對小趙：
- Week 1: 了解 Kubernetes 基本概念
- Week 2: 試著部署一個 Hello World
- Week 3: 分析優缺點
- Week 4: 產出報告

小趙需要的是「被拆解的任務」，而不是「開放式的挑戰」。

## 困難的平衡

當然，因材施教也帶來挑戰：

### 1. 會不會不公平？

有人質疑：「為什麼小陳可以自由發揮，我就要被管很細？」

我的回答：「因為你們處於不同階段。當你展現出足夠的自主能力，我也會給你同樣的自由。」

**公平不是一視同仁，而是按能力給予對應的責任。**

### 2. 如何避免標籤化？

我擔心「分類」會變成「標籤」，限制了他們的發展。

比如：「小王是需要推動型，所以他永遠都需要推動。」

所以我定期重新評估，給他們機會證明「我進步了」。

### 3. 管理成本很高

每個人用不同方式管，確實耗費更多時間和精力。

但我發現：**前期多花時間了解他們，後期管理反而更輕鬆。**

因為他們會變得更自主、更有產出。

### 4. 會不會有偏心？

我承認：面對不同性格的人，天生會有偏好。

我個人比較喜歡「自驅型」，因為管理起來輕鬆。

但我提醒自己：不能因為偏好而忽略其他人。

**每種類型都有價值，關鍵是找到適合他們的位置。**

## 我的管理矩陣

現在我用一個簡單的矩陣來思考：

```
         高自主 ─────────────► 低自主
高能力   │ 給舞台        │ 給方向
         │ (Senior)      │ (Mid)
         │               │
         ├───────────────┤
         │               │
低能力   │ 給支持        │ 給指引
         │ (有潛力的)    │ (Junior)
         └───────────────┘
```

- **高能力 + 高自主**：給舞台，讓他發揮
- **高能力 + 低自主**：給方向，幫他建立自信
- **低能力 + 高自主**：給支持，避免他亂闖
- **低能力 + 低自主**：給指引，手把手帶

當然，人不是固定的，會在這個矩陣中移動。

我的工作就是：幫助每個人往右上角移動（高能力 + 高自主）。

## 反思

管理最難的地方是：**沒有標準答案。**

同樣的方法，對 A 有效，對 B 可能完全無效。

這一年來，我學到最重要的事是：

**花時間了解每個人，比套用管理理論更重要。**

理論只是工具，但人才是核心。

當你真正理解他們的動機、焦慮、目標、性格，你就知道該如何幫助他們。

而當他們感受到「你真的在乎我的成長」，他們就會全力以赴。

**管理不是控制，而是賦能。**

而賦能的第一步，就是理解。
