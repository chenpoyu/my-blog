「Poyu！你來評評理，到底該用哪個方案？」

會議室裡，小陳和小王已經爭論了快半小時，兩人臉都紅了。

這是我當主管以來，第一次處理這麼激烈的技術衝突。

**我有點慌，但我知道我必須處理好，否則會影響整個團隊。**

## 衝突的起因

這次衝突是因為一個新專案的技術選型。

**小陳的立場（資深工程師）：**
「我們應該用 Spring Boot，團隊都熟悉，開發快，穩定，沒有風險。」

**小王的立場（Mid工程師）：**
「我們應該用 Quarkus，效能更好，啟動更快，未來趨勢。Spring Boot 太老了。」

**雙方的論點：**

小陳：
- 團隊熟悉 Spring Boot，學習成本低
- 生態系完整，遇到問題有很多資源
- 專案時程緊，不能冒險

小王：
- Quarkus 是雲原生框架，更適合容器化
- 效能比 Spring Boot 好很多（啟動快 10 倍）
- 我們應該學習新技術，不能一直用老東西

**兩人都有道理，但立場不同。**

而且，他們不只是技術爭論，還有情緒：

小陳覺得：「你一個 Mid，憑什麼質疑我的判斷？」  
小王覺得：「你只是資深，不代表你永遠對。」

**如果處理不好，會傷害兩人的關係，甚至影響團隊氣氛。**

## 我的第一反應：很想直接決定

當下我的第一反應是：「夠了，我來決定，就用 Spring Boot。」

這樣最快，爭論就結束了。

但我想起前主管跟我說過的話：

**「衝突不是壞事，代表大家在乎。重點是如何建設性地解決它。」**

所以我深呼吸，換個方式處理。

## 第一步：冷靜下來

我先讓會議暫停：

「我們先休息10分鐘，大家喝杯水，冷靜一下。」

然後我分別找小陳和小王聊。

### 跟小陳聊

**我：** 「你為什麼這麼堅持用 Spring Boot？」

**小陳：** 「因為這是商業專案，有時程壓力，我不想冒險。」

**我：** 「你覺得 Quarkus 的風險是什麼？」

**小陳：** 「團隊沒人會，學習會花時間，而且遇到問題不知道怎麼解。」

**我：** 「如果有人願意先花時間學，降低風險，你會考慮�as？」

**小陳：** 「那要看學到什麼程度...但說實話，我覺得沒必要。Spring Boot 已經夠好了。」

### 跟小王聊

**我：** 「你為什麼這麼想用 Quarkus？」

**小王：** 「因為我覺得 Spring Boot 太笨重了，而且我們一直在用老技術，學不到新東西。」

**我：** 「你了解 Quarkus 到什麼程度？」

**小王：** 「我自己玩過一個小專案，覺得很好用。」

**我：** 「如果用 Quarkus，你有信心在時程內交付嗎？」

**小王：** 「應該...可以吧。」（有點遲疑）

**我：** 「『應該』不夠，這是商業專案。如果你不確定，我們不能冒這個險。」

**小王：** 「但如果我們永遠不嘗試，永遠只用舊技術，我們會被淘汰的。」

### 我的觀察

分別聊過後，我發現：

**小陳的真正動機：**
- 表面：技術穩定性
- 深層：不想冒險，也不想被 Mid 質疑權威

**小王的真正動機：**
- 表面：技術優勢
- 深層：想學新技術，證明自己，也想挑戰資深工程師的權威

**這不只是技術爭論，還有情緒和人際關係的因素。**

## 第二步：找到共同目標

我把兩人叫回會議室：

**我：** 「我們先釐清一件事：我們的共同目標是什麼？」

**小陳：** 「把專案做好，準時交付。」

**小王：** 「嗯...對。」

**我：** 「所以，我們都同意：最重要的是『成功交付專案』，對吧？」

**兩人：** 「對。」

**我：** 「那技術選型只是手段，不是目的。我們應該選最適合這個專案的技術，而不是自己喜歡的技術，對嗎？」

**小陳：** 「對，所以應該選 Spring Boot。」

**小王：** 「我覺得 Quarkus 更適合...」

**我：** 「等等，我們先不爭論哪個適合。我們先定義『適合』的標準是什麼。」

## 第三步：建立評估標準

我在白板上寫下評估標準：

**1. 技術風險（40%）**
- 團隊熟悉度
- 遇到問題能否解決
- 社群資源是否充足

**2. 效能需求（30%）**
- 這個專案對效能的要求
- 現有技術能否滿足

**3. 未來維護（20%）**
- 客戶未來能否維護
- 技術生態的未來性

**4. 學習價值（10%）**
- 團隊能否學到新東西
- 對未來專案的幫助

**我：** 「這樣評分合理嗎？」

**小陳：** 「我覺得技術風險應該更高，50%。」

**小王：** 「我覺得效能需求應該更重要。」

**我：** 「好，那我們投票決定權重。」

最後大家同意：技術風險 45%、效能需求 30%、未來維護 15%、學習價值 10%。

## 第四步：數據說話

**我：** 「現在我們有標準了，但我們缺數據。」

**我提議：**
1. 小陳用 Spring Boot 做一個 POC（Proof of Concept）
2. 小王用 Quarkus 做一個 POC
3. 兩週後比較結果

**比較項目：**
- 開發時間
- 啟動時間
- 記憶體使用
- 吞吐量
- 遇到的問題

**小陳：** 「好，但兩週後如果 Quarkus 真的比較好，我也會承認。」

**小王：** 「沒問題！我會證明給你看。」

**兩人的眼神不再是對抗，而是競爭。**

## 兩週後的結果

POC 完成後，我們開會討論：

**Spring Boot（小陳）：**
- 開發時間：3 天
- 啟動時間：8 秒
- 記憶體：200 MB
- 吞吐量：2000 req/s
- 問題：沒有特別的問題

**Quarkus（小王）：**
- 開發時間：5 天（因為要學）
- 啟動時間：0.8 秒
- 記憶體：50 MB
- 吞吐量：2500 req/s
- 問題：有些功能文件不清楚，花時間摸索

**評分結果：**

| 標準 | 權重 | Spring Boot | Quarkus |
|------|------|-------------|---------|
| 技術風險 | 45% | 9/10 | 6/10 |
| 效能需求 | 30% | 7/10 | 9/10 |
| 未來維護 | 15% | 8/10 | 7/10 |
| 學習價值 | 10% | 5/10 | 9/10 |
| **總分** | | **7.75** | **7.25** |

**Spring Boot 小勝。**

### 做決定

**我：** 「從數據來看，兩者差不多，但 Spring Boot 在技術風險上更有優勢。考慮到這是商業專案，我決定用 Spring Boot。」

**小王：** 「我同意，雖然 Quarkus 效能更好，但開發時間確實比較長。」

**我：** 「但小王的觀察很對：我們確實需要學習新技術。所以我承諾：下一個適合的專案，我們用 Quarkus。而且由小王來主導。」

**小王：** 「真的嗎？」

**我：** 「真的。你已經證明 Quarkus 的價值，我們不應該忽視它。」

**小陳：** 「我也同意，這次POC讓我對 Quarkus 改觀了。雖然現在不適合，但未來可以考慮。」

**兩人握手。衝突解決。**

## 我學到的事

這次處理衝突，讓我學到很多：

### 1. 不要急著決定

第一反應「我來決定」看似有效率，但會埋下隱患。

花時間理解衝突背後的原因，才能真正解決問題。

### 2. 衝突背後通常有情緒

表面是技術爭論，實際可能是：
- 權威被挑戰（資深 vs 新人）
- 想證明自己（想學新技術）
- 恐懼改變（習慣舊技術）

要處理情緒，不只處理表面問題。

### 3. 建立評估標準

有標準，爭論才有意義。

否則就是各說各話，永遠沒結果。

### 4. 數據說話

POC 讓爭論從「我覺得」變成「數據顯示」。

這讓討論更客觀，也讓雙方都能接受結果。

### 5. 雙贏而非零和

我沒有讓一方完全勝利、另一方完全失敗。

而是：「這次用 A，下次用 B」，讓雙方都有收穫。

**這讓我深刻體會：**

**處理衝突的目的，不只是解決當下的問題，更是建立團隊的信任和文化。**

如果團隊成員覺得「這裡的爭論是建設性的」、「我的意見會被尊重」、「即使這次沒用我的方案，下次還有機會」，他們就會願意繼續提出想法。

**衝突不是壞事，代表大家在乎。**

重點是如何建設性地解決它。
