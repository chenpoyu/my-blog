## 請求進門，不等於可以直接執行

上一篇說了這個架構的背景和兩個核心概念：Order 和 Obligation。

使用者送出一個業務請求之後，
系統不是馬上開始執行，
而是先走幾道關卡，確認這件事「能不能做、行不行、准不准」。

這篇想把這幾道關卡講清楚。

## 第一層：Gateway 和 BFF

請求進來，最先接到的是 API Gateway。

Gateway 做的事情比較底層：
TLS 終止、JWT 驗證、Rate Limit、Trace ID 注入、入口路由。

它不碰業務邏輯。

過了 Gateway 之後，進到 BFF（Backend For Frontend）。

BFF 這一層，我覺得很容易被誤解定位。

它的工作是接前端請求、驗格式、萃取使用者上下文、準備好一個標準化的 Command，
然後交給後面的流程處理。

它不建合約、不建訂閱、不負責帳務，
也不決定要用哪個資料庫。

BFF 負責「準備」，不負責「履行」。

這個邊界如果沒守住，
BFF 很快就會變成什麼都做的大雜燴。

## 第一關：Catalog

BFF 整理好請求之後，
第一件事是去問 Catalog：

「這個 bundle 到底包含什麼？」

Catalog 是這套架構裡的產品目錄。

它知道每個 bundle 的組成，
哪些元件是必要的、哪些選填，
每個元件交給哪個 domain 執行，
以及這次套用的是哪個版本的目錄規則。

舉例，如果使用者申請「網路 + 電視」套餐：
Catalog 會說這個 bundle 包含網路合約、電視訂閱、設備分配三個元件，
分別由 contract、tv、asset 三個 domain 負責。

Catalog 的邊界很清楚：定義組成，不執行。

## 第二關：Eligibility

Catalog 確認了內容之後，
第二件事是確認這件事「業務上行不行」。

Eligibility 負責業務可行性：

- 地址在不在服務範圍
- 客戶有沒有申請資格
- 這個區域支不支援這個服務

這跟格式驗證不同，也跟權限驗證不同。

格式驗證是「你送的資料符不符合格式」，BFF 就可以擋。
權限驗證是「你被允不允許做這件事」，後面再說。
Eligibility 是「業務條件上，這件事能不能成立」。

三件事分開，才不會互相干擾。

## 第三關：Permission / OPA

確認業務可行之後，
才是確認「這個操作有沒有被允許」。

Permission 回答的是：

這個使用者、這個角色、在這個公司範圍內，
能不能做這個動作？

通常這裡會搭配 OPA（Open Policy Agent）這類工具，
把政策定義和業務邏輯分開管理。

政策不是寫死在 service 裡，
而是由獨立的地方管，有版本、有紀錄，可以稽核。

Permission 說不行，直接回拒絕，不進訂單流程。

## 為什麼不把三關合成一關

我知道有人看到這裡會想：

Catalog、Eligibility、Permission，
這樣分是不是太細了？一個 check service 搞定不就好？

因為這三件事的關注點完全不同。

Catalog 知道「產品長什麼樣子」。
Eligibility 知道「業務條件行不行」。
Permission 知道「這個人能不能做」。

如果混在一起，
改政策可能動到 eligibility 邏輯，
改產品可能影響權限判斷，
要查稽核紀錄也找不到乾淨的邊界。

分開是為了讓各自能獨立演進，
也是為了出問題時能快速定位是哪關卡住。

## 通過之後才建 Order

三關都通過，
系統才真正建立 Order，
然後進入下一個階段。

Order 建起來之後，接下來就是把它拆解成多個 Obligation，
再決定每個 Obligation 要送去哪裡執行。
