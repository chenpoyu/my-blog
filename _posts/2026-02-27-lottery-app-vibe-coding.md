## 緣起：公司尾牙要一個「不一樣」的抽獎

事情是這樣的。2026 是丙午馬年，我家人要辦尾牙，說不想用那種爛大街的隨機抽號碼。

「有沒有那種⋯⋯很有氣氛的？最好像廟會那樣，有命理、有火焰、有馬在跑的？」

好，你都說了，那就做吧。

## 需求長什麼樣

整理一下他們零散丟出的需求：

1. **四輪抽獎**，每輪有不同主題和獎項名稱
2. 前兩輪用一份名單（中獎的人自動移除），後兩輪用另一份名單（可以重新洗牌）
3. 抽獎過程要**有儀式感**——最好有名字滾動的鼓聲效果，然後一個一個翻牌揭曉
4. 每個中獎者要附一段「命理鑑定語」，像廟裡抽籤那樣，增加歡樂氣氛
5. 支援 CSV 匯入，HR 那邊可以直接把 Excel 名單丟進來
6. 手機也要能看（現場要投影，但大家也想用手機刷）

額外我自己加的想法：既然馬年，那就搞個**五行命理引擎**，根據姓名筆畫和出生年份算出天干地支、五行屬性，再拿來當抽獎加權。不是真的算命，但會讓鑑定語看起來超像真的。

## 技術選擇

這種一次性的活動工具，最重要的就是**快**跟**花**。

- **React 19 + Vite 7**：開發快、打包快、部署快
- **Framer Motion**：動畫是這個 App 的靈魂，卡牌翻轉、彈簧彈出、淡入淡出全靠它
- **SASS**：1,300 行的樣式，用純 CSS 寫我會瘋掉
- **Lucide React**：圖示庫，簡單乾淨
- 沒有後端，純前端搞定。名單從 CSV 匯入或手動貼上，資料存在 state 裡就夠了

## 我怎麼開發的 — 聊聊 Vibe Coding

這個專案從零到部署，大約花了一個下午。

老實說，這次我是用 **Vibe Coding** 的方式做的。什麼意思？就是我先在腦子裡有個畫面——紅金色的廟會風格、火焰在底部跳動、一匹馬在背景奔跑、名字像跑馬燈一樣滾、最後翻牌揭曉——然後把這些 vibe 餵給 AI，讓它幫我把骨架先搭出來。

流程大概是這樣：

1. 我先把需求和畫面想像用文字描述出來，丟給 AI
2. AI 吐出一版基礎的元件結構和樣式
3. 我拿回來看，覺得「差不多但不夠」的地方就自己改
4. 比較吃設計感的部分（像火焰粒子、卡牌翻轉的 timing、鑑定語的文案）我自己手動調

好處很明顯：**省掉大量的 boilerplate 時間**。像 CSV 解析、表單處理、基礎的 state 管理這些，AI 生出來的東西可以直接用，不用從頭敲。

但也有很明確感受到的限制：

- **設計品味 AI 是沒有的**。它可以寫出「一個火焰動畫」，但那個火焰好不好看、跳動的節奏對不對味，這些全靠自己調。1,300 行 SCSS 裡面大概有一半是我反覆調整 timing、顏色漸層、模糊度的結果。
- **命理鑑定語是我自己寫的**。四輪 × 八種模板 = 32 段鑑定語，每段都要搭配五行屬性和生肖，寫出那種「半認真半搞笑」的算命口吻，這種事 AI 寫出來的要嘛太正經、要嘛太尬，最後還是自己操刀。
- **互動流程的微調全是手動**。像是「滾動名字多久才停」、「翻牌之間的鼓聲要滾多久」、「彩帶要撒幾秒」，這些影響使用體驗的細節，全部都是我自己一次次跑、一次次改出來的。

所以我的看法是：Vibe Coding 很適合**快速打地基**，但**最後讓這東西好用、好看、有記憶點的，還是要靠人去打磨**。AI 幫你從 0 到 60，你自己從 60 推到 90。

## 這個 App 的重點設計

### 四輪抽獎機制

這不是簡單的「按一下隨機選人」。四輪之間有連動關係：

| 輪次 | 主題 | 名單邏輯 |
|------|------|----------|
| 第一輪 🔥 | 離火微光・五行補運獎 | 從名單一全員抽，中獎者移除 |
| 第二輪 🙏 | 神光普照・天后契子獎 | 從第一輪剩餘的人裡抽 |
| 第三輪 🏆 | 丙午金匱・赤馬銜金獎 | 切到名單二，全員重新參加 |
| 第四輪 🌟 | 水火既濟・特別獎 | 名單二扣除第三輪得主 |

前兩輪跟後兩輪可以用不同名單，這是因為公司有些獎項是全員參加、有些是限定部門。這個「雙名單系統」在設定畫面就能分開匯入。

### 五行命理引擎

這是整個 App 最好玩的部分。每個人抽獎時，系統會算出他的「命理數據」：

- **姓名筆畫**：查一張內建的漢字筆畫表（涵蓋百家姓 + 常用名字用字，查不到的就用 Unicode 碼推算）
- **筆畫轉五行**：尾數 1-2 木、3-4 火、5-6 土、7-8 金、9-0 水
- **出生年天干地支**：用 1984 甲子年為基準推算
- **2026 丙午年親和度**：火年嘛，火最旺 (100)、木生火次之 (90)、土洩火再次 (80)、金被火剋 (50)、水剋火最低 (40)

最後抽獎的權重是：**姓名五行 50% + 生日命格 20% + 隨機值 30%**。

隨機值佔 30% 是刻意的——不能真的讓命理決定一切，不然就變成「出生年決定中獎」了，那太荒謬。但有了五行加權，就能讓鑑定語講出來的東西跟人名、生肖對得上，看起來很像真的在算。

### 逐一揭曉動畫

這是現場氣氛的關鍵。抽完之後不是直接列出所有得獎者，而是：

1. 先出現一個蓋板，上面寫「天命感應中⋯」
2. 名字開始快速滾動（像老虎機那樣）
3. 停下來，翻牌，顯示得獎者姓名 + 生日 + 專屬鑑定語
4. 按「下一位」，再來一次鼓聲滾動 → 翻牌
5. 最後一位揭曉後，全部得獎者以卡片牆的方式展示

第三輪和第四輪是大獎，揭曉時還會加上彩帶特效跟光暈動畫。

### 32 段命理鑑定語

四輪各 8 種模板，會根據中獎者的姓名五行、生肖、天干地支動態填入。舉幾個例子：

> 【補運】姓名「王大明」行走 15 畫屬土。2026 丙午年火旺，你這格正好「火水既濟」，不需媽祖操心，自帶小暖爐逢凶化吉。

> 【銜金】44人唯一天選！李小花姓名 16 畫「火中煉金」。在丙午年這把熊熊烈火中，你就是那塊能承載財富的真金！

> 【既濟】蒸鍋至尊！張志豪屬牛，筆劃 27 完美中和火燥。今日得獎象徵「水火平衡」，保佑全家馬年蒸蒸日上。

每段都用了那種廟裡師公講話的半文半白口吻，現場念出來效果很好。

### 廟會風格 UI

整個 App 的視覺設計圍繞幾個元素：

- **底色**：深紅到暗紅的放射漸層，像廟宇大殿
- **金色系**：文字、邊框、按鈕全用金色調，搭配 Noto Serif TC 襯線字體
- **火焰粒子**：底部 12 朵火焰，用 CSS 做跳動動畫，不同 delay 錯開，看起來像一排蠟燭
- **浮動粒子**：20 顆金色/橘色/紅色粒子從底部往上飄
- **奔馬動畫**：一匹馬的 PNG 圖從畫面左邊跑到右邊，循環播放
- **每輪不同背景圖**：四輪各有一張專屬的全版背景圖（AI 生成的圖），配合主題氛圍切換

### CSV 匯入

HR 不會用開發工具，所以匯入要做到最簡單：

- 點一個按鈕、選檔案、搞定
- 自動偵測並跳過標題列（判斷第一行是不是有「姓名」「日期」之類的字）
- 自動去 BOM（Excel 另存 CSV 一定會帶的那個 `0xFEFF`）
- 支援 `YYYY-MM-DD` 和 `YYYY/MM/DD` 兩種日期格式
- 沒有生日也沒關係，純姓名照樣能抽

## 專案規模

整個專案約 2,500 行程式碼（不含 `node_modules` 和圖片）：

- SCSS 樣式：約 1,430 行（佔大宗，廟會風格需要大量手刻）
- React 元件：約 830 行
- 命理引擎 + 資料：約 300 行

三次 commit 就搞定了，一天完工。

## 結語

這是一個很典型的「需求明確、時限很短、注重體驗」的小專案。Vibe Coding 在這種場景下發揮得很好——讓我把時間花在**調動畫節奏、寫鑑定語文案、打磨 UI 細節**這些真正影響現場體驗的事情上，而不是卡在 CSV 解析或 state 管理這種工程雜活。

最後活動當天用起來，同事們反應很好。尤其是鑑定語唸出來的時候，全場笑翻。有人還特地來問「這個命理是真的嗎？」

不是。但如果你信的話，也可以是。
