## 三個月的轉換旅程

從五月中決定轉換到 React，到現在已經三個月。期間我們：
- 完成 3 個政府專案的後台系統
- 建立一套 Config-Driven 的開發框架
- 團隊 5 個人全部上手 React

這段旅程不只是技術轉換，更是團隊學習與協作的過程。

## 技術層面的收穫

### 1. 從 Options API 到 Hooks

一開始最不習慣的就是 React Hooks。Vue 的 Options API 結構清晰，`data`、`methods`、`computed` 各司其職。React 卻把所有邏輯都放在函式裡。

但用久了會發現，Hooks 的彈性更高：
- 邏輯重用更直覺（自訂 Hook）
- 不需要 `this`，減少心智負擔
- 函式組合的概念更純粹

現在回頭看 Vue，反而覺得有點「框框」太多。

### 2. 虛擬 DOM 的差異

Vue 和 React 都用虛擬 DOM，但實作邏輯不同：
- Vue 依賴 template 編譯優化，效能預設就不錯
- React 更依賴開發者手動優化（memo、useMemo）

這讓我們更深入理解「為什麼會重新渲染」，也學會用 Profiler 找效能瓶頸。

### 3. 生態系的選擇

React 的生態系更分散，沒有官方的「全家桶」。起初覺得混亂，後來發現這反而是優勢：
- 可以根據專案需求選擇工具
- 不被框架綁死
- 學習其他技術時更容易遷移

我們最終選擇的技術棧：
- **UI**：Ant Design（政府專案首選）
- **路由**：React Router v6
- **表單**：Ant Design Form + 自訂 Hook
- **狀態管理**：Context + useState（夠用了）
- **HTTP**：Axios

沒有使用 Redux / MobX，因為後台系統的狀態管理需求不複雜。

## 架構設計的心得

### Config-Driven 的成功與限制

Config-Driven 的 CRUD 系統確實大幅提升開發效率：
- 標準頁面從 2 小時縮短到 10 分鐘
- 程式碼一致性高，維護容易
- 新成員上手快

但也有限制：
- 學習成本較高（需要理解配置結構）
- 複雜頁面反而變得彆扭
- 除錯時需要追蹤通用元件邏輯

我們的經驗是：**70% 標準化，30% 保留彈性**。

### 從後端思維學到的

Config-Driven 的靈感來自後端的「配置驅動」：
- Spring Boot 的 `application.yml`
- Hibernate 的 Entity 配置
- RESTful API 的統一規範

這讓我重新思考：**前後端在架構設計上有很多共通之處**。

好的架構不分前後端，都要追求：
- 減少重複程式碼
- 提高可維護性
- 在通用性與彈性間取得平衡

## 團隊協作的挑戰

### 1. 學習曲線

團隊成員對 Vue 都很熟悉，轉到 React 的學習曲線：
- **第 1 週**：看文件、寫範例（挫折感很重）
- **第 2-3 週**：能完成簡單頁面，但常卡關
- **第 4 週後**：逐漸掌握 Hooks 的用法
- **第 8 週後**：可以獨立設計元件架構

關鍵是**邊做邊學**，不要花太多時間在教學上。

### 2. Code Review 很重要

轉換初期，大家寫出來的 React 程式碼風格差異很大。透過 Code Review 我們逐漸建立共識：
- 元件命名規範
- Hook 的使用原則
- 何時該拆分元件
- 效能優化的時機

Code Review 不只是找 bug，更是團隊學習與統一風格的過程。

### 3. 文件化很必要

我們建立了內部的「React 開發指南」，包含：
- 專案結構說明
- 常用 Hook 範例
- Config-Driven 系統使用手冊
- 效能優化 checklist

有了這份文件，新成員上手速度快很多。

## 與 Vue 的對比

很多人問：**React 比 Vue 好嗎？**

我的答案是：**看情境**。

| 面向 | Vue | React |
|-----|-----|-------|
| 學習曲線 | 較平緩 | 較陡峭 |
| 開發速度 | 快（預設最佳化） | 需要更多手動優化 |
| 彈性 | 中等 | 高 |
| 生態系 | 官方全家桶 | 社群分散 |
| 適合場景 | 中小型專案、快速開發 | 大型專案、需要高彈性 |

對我們來說，React 的選擇是**被動的**（因政策考量），但過程中學到很多。

## 如果重來一次

如果再做一次 Vue 轉 React，我會：

1. **先建立最小可行原型（MVP）**：先做一個簡單頁面，跑通整個流程
2. **先不追求完美架構**：邊做邊重構，不要一開始就想設計完美系統
3. **每週做技術分享**：讓團隊互相學習，減少重複踩坑
4. **保留更多彈性**：不要太執著於「統一框架」

## 對未來的影響

這三個月的轉換，對我最大的影響是：

**不再把自己侷限在某個框架或技術**。

Vue 和 React 都只是工具，重要的是背後的設計思維：
- 元件化
- 狀態管理
- 效能優化
- 架構設計

這些概念是跨框架、跨語言的。

## 結語

從 Vue 轉到 React 的這段旅程，技術是其次，更重要的是：
- 團隊如何在變化中保持產出
- 如何在學習與交付間取得平衡
- 如何建立可擴充的系統架構

接下來會繼續分享 React 的進階應用，以及其他技術領域的實戰經驗。

技術的學習永遠沒有終點，但每一次轉換都是成長的機會。
