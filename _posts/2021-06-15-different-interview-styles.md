持續面試的過程中，我發現面試資深和資淺工程師，真的需要完全不同的策略和問題設計。

上週同一天面試了兩位候選人，一位是剛畢業的新鮮人，另一位是有八年經驗的資深工程師。同樣的問題問下去，得到的反應完全不同，也讓我重新思考面試的方法。

## 面試資淺工程師的重點

對於剛畢業或一到兩年經驗的工程師，我現在比較重視這幾點：

**基本語法和觀念**  
問一些基礎題，像是 Java 的繼承、多型，或是資料結構的基本概念。不用太難，但要確保基礎是扎實的。有些新鮮人連 interface 和 abstract class 的差異都說不清楚，這就有問題了。

**解決問題的思路**  
我會給一個簡單的演算法題，像是反轉字串或找陣列中的最大值，不看重他能不能寫出最優解，而是看他的思考過程。會不會先問清楚需求？會不會考慮邊界條件？遇到卡住的時候怎麼辦？

**學習熱忱**  
問他最近在學什麼，為什麼學，怎麼學的。有些人會說「老師教的」，有些人會說「我看到某個開源專案很有趣，自己研究了一下」。後者通常比較有潛力。

## 面試資深工程師的重點

資深工程師就完全不同了，技術只是基本門檻，更重要的是：

**系統設計能力**  
我會給一個實際的場景，比如「設計一個電商網站的訂單系統」，然後看他怎麼拆解問題。他會考慮資料庫設計嗎？會想到併發處理嗎？會提到快取策略嗎？從這裡可以看出他的技術廣度。

**過往專案經驗**  
深入問他過去做過最複雜的專案，遇到什麼挑戰，怎麼解決的。如果他只是說「我們用了 Spring Boot」，那還不夠；如果他能說「當時遇到效能瓶頸，我們透過分析發現是 N+1 查詢問題，後來改用 JOIN 和快取機制優化」，這就是我要的深度。

**帶人經驗**  
如果是要找技術主管或 Senior，我會問他有沒有帶過人。怎麼教新人？怎麼做 Code Review？遇到能力不足的組員怎麼處理？這些軟實力在資深職位非常重要。

**技術廣度與深度**  
資深工程師通常會有自己專精的領域，但也要有一定的廣度。我會問他除了日常工作的技術棧，還接觸過什麼？有沒有關注新技術？如何評估要不要導入新技術？

## 印象深刻的資深候選人

有個資深候選人讓我印象特別深刻。當我問他「如果要設計一個高併發的訂單系統，你會怎麼做？」

他沒有直接回答，而是先反問我：「請問預期的 QPS 是多少？使用者分布在哪些地區？對一致性的要求到什麼程度？」

然後他才開始說：「如果是這樣的話，我會考慮用訊息佇列來削峰，資料庫採用讀寫分離，關鍵資料用 Redis 快取...」而且他會主動說明每個選擇的 trade-off。

這種雙向交流，才是資深面試該有的樣子。他不只是回答問題，而是在討論解決方案。

最後我們錄取了他，現在他負責其中一個政府專案的技術架構，表現很不錯。

## 反思

面試不同層級的工程師，真的要用不同的標準。用面試資深的方式去面資淺，會嚇跑很多有潛力的新人；用面試資淺的方式去面資深，又看不出真正的實力。

找到適合的評估方式，才能選到對的人。
