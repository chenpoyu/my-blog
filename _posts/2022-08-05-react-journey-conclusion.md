## 三個月的總結

從五月中旬決定從 Vue.js 轉換到 React，到現在八月初，整整三個月。

這段期間我們：
- 建立一套 Config-Driven 的 CRUD 框架
- 導入測試與 CI/CD 流程
- 團隊全員上手 React 開發

更重要的是，這次轉換讓我對「技術選型」、「架構設計」、「團隊協作」有了更深的思考。

## 關於技術選型

### 被動選擇也能有主動收穫

這次轉換的起因是**外部政策壓力**，而非主動選擇。一開始團隊士氣有點低落，覺得是「被迫」學新東西。

但過程中我們發現：**即使是被動的轉換,也能創造主動的價值**。

我們沒有把它當成「不得不做的事」,而是當成「重新檢視開發流程的機會」：
- 既然要重寫，就順便優化架構
- 既然要學新技術，就建立更好的開發規範
- 既然要投入時間，就把測試與 CI/CD 一起做好

結果反而比單純「維持現狀」獲得更多。

### 技術本身不是重點

Vue 好還是 React 好？這個問題我們爭論過很多次。

最後的結論是：**都好，也都不夠好**。

- Vue 的開發體驗很好，但彈性稍嫌不足
- React 的彈性很高，但學習曲線較陡
- 兩者在效能、生態系、社群支援上都很成熟

真正影響專案成敗的，不是用哪個框架，而是：
- **團隊對技術的掌握程度**
- **架構設計的合理性**
- **開發流程的完整性**

一個用 Vue 寫得很糟的專案，不會因為改用 React 就變好。

## 關於架構設計

### 從「寫程式」到「設計系統」

這次最大的收穫，是從「寫程式」進化到「設計系統」。

過去我們寫前端，專注在「這個功能怎麼實作」。現在更關注：
- 如何減少重複程式碼
- 如何讓新成員快速上手
- 如何在未來擴充時不用大改

Config-Driven 的 CRUD 框架就是這種思維的產物。

### 好的抽象需要時間沈澱

一開始我們想「一步到位」設計完美的框架，結果發現：
- 想太多，遲遲做不出來
- 考慮太多邊界情況，反而讓系統變複雜
- 預設的「通用性」根本用不到

後來我們改變策略：
1. 先解決眼前的問題
2. 重複出現時再抽象化
3. 有三個以上相同需求才做通用元件

**Rule of Three**：重複三次再抽象，而非一開始就抽象。

### 彈性與規範的平衡

React 的彈性是雙面刃：
- 好處：可以用任何方式解決問題
- 壞處：每個人寫法都不一樣

我們的做法是：
- **70% 標準化**：大部分情況用統一框架
- **30% 保留彈性**：複雜場景允許客製化

不追求「所有東西都統一」，而是「讓簡單的事情保持簡單」。

## 關於團隊協作

### Code Review 比想像中重要

以前我們很少做 Code Review，覺得浪費時間。

這次轉換過程中，Code Review 變成最重要的學習管道：
- 看到別人的寫法，學習新技巧
- 討論為什麼這樣寫，理解設計思路
- 統一團隊風格，減少認知負擔

現在我們每個 PR 都至少要一個人 Review，雖然花時間，但長期來看很值得。

### 文件化是投資而非成本

一開始覺得寫文件很麻煩，後來發現：
- 新成員上手速度快很多
- 減少重複回答問題
- 自己隔一段時間也會忘記，文件是備忘錄

我們建立了：
- **開發指南**：專案結構、命名規範、常用 Hook
- **Config 範例**：各種頁面類型的配置檔範例
- **Troubleshooting**：常見問題與解決方式

這些文件的投資報酬率非常高。

### 允許犯錯，但要記錄學習

轉換過程中，每個人都踩了很多坑：
- `useEffect` 無限迴圈
- 忘記清理 event listener
- 過度優化導致程式碼複雜

我們的做法是：
- 在週會分享「這週踩的坑」
- 把常見錯誤寫進開發指南
- 鼓勵提問，不要默默苦惱

允許犯錯，但要確保不重複犯錯。

## 如果重來一次

如果再做一次技術轉換，我會：

1. **更早建立 MVP**：不要想太多，先做出一個能動的版本
2. **更頻繁的技術分享**：每週固定時間討論新發現
3. **更早導入測試**：不要等「穩定了再補測試」
4. **更積極的實驗**：不怕失敗，多嘗試不同做法

最重要的是：**保持學習心態，把轉換當成成長機會**。

## 對未來的影響

這三個月的經驗，改變了我對技術的看法：

### 1. 框架會變，原理不變

Vue、React、Angular 會不斷演進，甚至被取代。

但背後的原理：
- 元件化思維
- 狀態管理
- 效能優化
- 架構設計

這些是跨框架、跨語言的。

### 2. 廣度與深度要並重

過去我專注在「深度」：把 Vue 學到很精。

現在更注重「廣度」：理解不同框架的設計哲學，才能做出更好的選擇。

不是要「什麼都學」，而是「了解不同思維方式」。

### 3. 技術服務於目標

技術是工具，不是目的。

我們轉換到 React 的目的是：
- 降低政策風險
- 建立更高效的開發流程
- 提升團隊技術能力

如果沒有達成這些目標，技術轉換就只是「為了轉換而轉換」。

## 給正在轉換技術的你

如果你也在經歷技術轉換，我想說：

1. **轉換過程會很痛苦**，但熬過去會很有收穫
2. **團隊的支持很重要**，不要一個人硬撐
3. **允許自己犯錯**，學習曲線是正常的
4. **專注在目標上**，不要被細節困住
5. **享受過程**，學習新技術本身就很有趣

## 結語

從 Vue.js 到 React 的這段旅程，技術轉換只是表面。

真正的收穫是：
- 重新審視開發流程
- 建立更好的團隊協作方式
- 培養「設計系統」而非「寫程式」的思維

接下來會繼續分享其他技術領域的經驗，包括後端、雲端、架構設計等。

技術的學習是一場馬拉松，不是短跑。

重要的不是「現在會什麼」，而是「能不能持續學習」。
