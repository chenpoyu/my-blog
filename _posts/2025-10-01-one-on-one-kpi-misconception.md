## 又到了這個時間

每年 Q4 就是這樣，一邊要收尾今年的專案，一邊要開始訂明年的 KPI。身為部門主管，這段時間最主要的工作之一就是跟每個人坐下來一對一聊。

聊今年做了什麼、哪些做得不錯、哪些卡住了，然後開始談明年的方向跟目標。

今年排下來大概要跟十幾個人聊，每個人至少半小時到一小時。體力活。

## 聊完幾個人之後發現一件事

前幾場聊完，我發現一個蠻普遍的問題——**很多人其實不太知道 KPI 到底要怎麼寫**。

不是說他們不認真。每個人都有想法，也都有想做的事。但寫出來的東西，常常會變成這樣：

- 「今年要多讀幾本技術書」
- 「學會一個新框架」
- 「希望可以加薪」
- 「想去考一張證照」
- 「保持身體健康，多運動」

看到這些的時候，我不會直接說「這不對」。因為這些作為個人目標都很好，問題是它們不是 KPI。

## KPI 跟「個人願望清單」的差別

我跟同仁解釋的方式通常很直白：

**KPI 是你跟公司之間的約定，不是你跟自己的新年新希望。**

「今年要讀五本書」——這是個人成長計畫，很好，但公司沒辦法衡量這件事對業務的幫助是什麼。

「把 A 模組的回應時間從 800ms 降到 300ms」——這就是 KPI，因為它**具體、可量化，而且跟部門目標直接相關**。

差別在哪？在於這件事做完之後，**除了你自己，還有誰會因此受益**。如果答案只有「我自己」，那它比較像是個人目標。如果答案是「我的團隊」、「我們的產品」、「我們的客戶」，那它才比較像是 KPI。

## 常見的幾種「偽 KPI」

聊了一輪下來，我大概歸納出幾種常見的狀況：

### 1. 太模糊，沒有辦法衡量

> 「提升自己的技術能力」

這句話誰都會說，但做到什麼程度算「提升」了？你怎麼知道年底的時候你有沒有達成？主管又怎麼幫你評？

改成：「Q2 前完成 Redis 快取架構的導入，讓報表查詢 API 的 P95 latency 從 1.2s 降到 400ms 以內」。一樣是在提升技術能力，但現在有個明確的標的了。

### 2. 跟部門目標搭不上

> 「學會 Rust」

如果部門技術棧是 .NET + React，你學 Rust 很好，但這件事跟團隊今年要交付的東西沒有直接關係。放在個人學習計畫裡沒問題，但放在 KPI 裡就有點奇怪。

KPI 要能**往上對齊**——你的 KPI 加起來，應該要能撐起部門的目標；部門的目標加起來，要能撐起公司的目標。這條線要連得起來。

### 3. 只有「做了」，沒有「做到什麼程度」

> 「負責 B 專案的後端開發」

嗯，然後呢？做完就算達成嗎？做完但 bug 一堆算達成嗎？做完但延遲三個月算達成嗎？

KPI 要有結果導向的衡量標準。同樣是負責後端開發，可以寫成：「Q3 前完成 B 專案後端 API 開發並上線，上線後一個月內 production bug 不超過 3 件」。

### 4. 把「過程」當成「目標」

> 「每週參加讀書會」

參加讀書會是手段，不是目的。你參加讀書會是為了什麼？如果是為了讓 code review 的品質提升，那 KPI 應該寫在 code review 的品質上，讀書會只是你達成的方式之一。

## 我自己怎麼引導

一對一面談的時候，我通常不會直接幫他們寫 KPI，而是反問幾個問題：

1. 「你覺得明年部門最重要的三件事是什麼？」——先拉齊方向
2. 「這三件裡面，你可以扛哪一塊？」——找出他的角色
3. 「扛這一塊的話，做到什麼程度算做好了？」——逼出具體標準
4. 「這個標準，年底的時候你自己能不能量出來？」——確認可衡量

通常問完這四題，東西就八九不離十了。剩下就是數字調一調、時程對一對。

## 寫在最後

其實我能理解為什麼大家會寫出「讀書、學習、考證照」這種東西。因為在學校或自己做計畫的時候，目標本來就是這樣訂的。沒有人教過你，進了公司之後「目標」這兩個字的意思不太一樣。

所以這不是誰的錯，就是一個需要被說清楚的事。

這一輪面談聊下來蠻花時間的，但我覺得值得。比起年底才發現大家的 KPI 根本沒對齊，現在花時間把觀念拉齊，後面省的力氣絕對更多。

下週會接著做第二輪，確認每個人修完之後的版本。到時候應該會把一些具體的寫法整理出來給大家當參考。
