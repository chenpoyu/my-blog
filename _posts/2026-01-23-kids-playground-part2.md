## 接續上回

[上一篇]({% post_url 2026-01-16-kids-playground-part1 %})講了整體架構跟顏色配對遊戲。這篇接著聊其他幾款遊戲的設計，以及讓小朋友「實測」之後踩到的坑。

做大人用的系統，你可以寫使用說明、放 tooltip、做引導流程。但做給兩三歲的小孩？他們不會看說明，也不會讀提示文字，所以**遊戲本身就得夠直覺**，看一眼就知道要幹嘛。

這個原則在後來的每一款遊戲裡都被反覆驗證。

## 動物配對：重做了一次的教訓

第一版的動物配對，我做的是「影子配對」——左邊是彩色的動物圖案，右邊是灰色剪影，你要把動物拖到對的影子上。

聽起來合理對吧？結果幼兒園的小朋友完全不知道在玩什麼。

他們看著灰色的影子，問老師：「這個是什麼？」然後就開始亂點。影子太抽象了，emoji 本身的輪廓在縮小成灰色剪影之後，根本看不出是什麼動物。

反省了一下，決定砍掉重練。

第二版改成了**記憶翻翻樂**——跟顏色配對一樣的翻牌機制，但卡片上放的是動物 emoji。12 張卡片（6 對動物），4×3 的格子排列，手機上自動變成 3 欄。

每張卡片背面是一片薄荷綠加上 🌿 的圖案，翻開之後會看到動物 emoji 和中文名稱。配對成功的卡片會亮綠色邊框，而且有一個 `pulse` 動畫的視覺回饋。

這個版本小朋友一看就懂了：「翻開找一樣的！」

教訓就是：**大人覺得好玩的不一定適合小孩**。影子配對在邏輯上很有趣，但在視覺辨識上對兩三歲的小孩太難了。記憶翻翻樂雖然機制簡單，但小孩實際上就是吃這套。

卡片翻轉的動畫用了 CSS 的 `perspective` 和 `rotateY`，配合 `backface-visibility: hidden` 做出前後面的效果。翻轉的 timing function 用 `cubic-bezier(0.68, -0.55, 0.265, 1.55)` 這個彈跳曲線，讓翻牌的感覺有一點 Q 彈，小孩看到會覺得有趣。

## 數字氣球：那個重疊的 bug

數字氣球的規則是：天空中飄著 7 顆氣球，上面分別寫了 1 到 7。你要按照順序去戳——先戳 1、再戳 2，以此類推。

第一版的氣球位置是完全隨機的：

```javascript
x: 10 + Math.random() * 70
y: 15 + Math.random() * 55
```

上線之後馬上就被發現問題——有時候兩顆氣球會重疊在一起，數字被擋住看不到。小孩就更慌了，明明找不到下一個數字，氣球又飄來飄去。

解法是改用**格子定位**。把畫面切成 3×3 的虛擬格子，每顆氣球分配到一個格子內，再加上一點隨機偏移讓它看起來不是太整齊：

```javascript
const cols = 3, rows = 3
for (let r = 0; r < rows; r++) {
  for (let c = 0; c < cols; c++) {
    positions.push({
      x: 12 + c * 30 + (Math.random() * 14 - 7),
      y: 8 + r * 28 + (Math.random() * 10 - 5),
    })
  }
}
```

9 個格子裡隨機挑 7 個，氣球的大小範圍也從 `0.85~1.2` 收窄到 `0.9~1.1`，進一步降低重疊的機率。

改完之後重疊的狀況基本消失了。但看起來還是有自然的隨機感，不會太呆板。

每顆氣球還有飄浮動畫——用 CSS 的 `@keyframes` 讓它上下輕微晃動，配上不同的 `animation-delay`，視覺上會像真的在天空中飄。戳破的瞬間會播放 `playPop()` 音效（隨機高頻正弦波），然後氣球縮小消失、爆出一個 💥。

## 形狀排排看：分兩區的設計

這款遊戲的概念是認識形狀。畫面分成上下兩區：

- **上方是彩色形狀**：圓形、正方形、三角形、星形、愛心、菱形，每個都有自己的顏色
- **下方是灰色輪廓**：同樣的形狀，但只有線條，順序被打亂

你先點上面的彩色形狀，再點下面對應的輪廓，配對成功輪廓就會填上顏色。

形狀全部用 SVG 手繪——圓形用 `<circle>`、正方形用 `<rect>`、三角形用 `<polygon>` 的三個點座標。因為是 SVG，放大縮小都不會糊，在不同螢幕尺寸上效果都一樣好。

有個小細節是選擇狀態的視覺回饋。當你點了一個彩色形狀之後，它會有明顯的放大效果跟光暈，同時下方的輪廓也會顯示「可以點擊」的 ready 狀態。這種兩步驟的操作（先選再放）對小孩來說不算太難，但比起單純的翻牌還是進階了一些，所以建議年齡標了 3+。

錯誤回饋也很溫和——放錯位置的話，輪廓會晃一下加上變紅，半秒後就恢復。不會彈出「答錯了！」之類的訊息打擊信心。

## 共用元件：WinModal 的設計考量

四個遊戲都用了同一個 `WinModal` 過關彈窗，值得單獨講一下。

當遊戲完成時，會觸發幾件事：

1. 播放 `playWin()` 過關音效（C5 到 C6 的八音音階）
2. `canvas-confetti` 從畫面中央撒出彩色碎紙
3. 彈出 WinModal，顯示大獎杯 🏆、星星評分、鼓勵文字

星星用 `popIn` 動畫一顆一顆彈出來（延遲 0.3s、0.5s、0.7s），這個節奏感小孩很喜歡。

底下兩顆按鈕：「🔄 再玩一次」和「🏠 回首頁」。按鈕做很大、圓角很圓，顏色也鮮明（珊瑚紅跟薄荷綠），確保小手按得到。

有一個 RWD 的坑：原本按鈕是橫排的，在手機上很容易按錯。後來加了 mobile 斷點，320px 以下的螢幕按鈕改成垂直排列、寬度撐滿，好按多了。

```scss
@include mobile {
  width: 100%;
  flex-direction: column;
}
```

## 首頁：小孩也要看得懂

首頁的設計花了一些心思。

最上面是標題「歡樂小遊戲樂園」，每個字都有不同的顏色，而且有 `bounceIn` 進場動畫，一個接一個彈出來。背景有飄浮的雲朵、星星、蝴蝶當裝飾。底部是一排小花小草。

整個視覺就是一個明亮歡快的感覺，小孩看到會覺得「這裡有好玩的東西」。

遊戲卡片做成大大的方塊，上面放 emoji 圖案、遊戲名稱、簡短說明、建議年齡。每張卡片有自己的背景漸層色，hover 的時候會浮起來、emoji 會放大旋轉、還有一道光澤掃過去的 `shine` 效果。

在手機上是 2 欄排列，每張卡片都是正方形（`aspect-ratio: 1`），手指好點、資訊也足夠。

## 讓 AI 幫忙但品質要自己顧

這四款遊戲的程式碼，有很大比例是靠 Vibe Coding 產出的。但我想強調的是，AI 幫你生成程式碼只是第一步，真正花時間的是後面的調整。

舉幾個例子：

- AI 生成的氣球位置用純隨機，看起來沒問題但實測會重疊，要自己想格子定位的解法
- AI 生成的翻牌動畫太快，小孩翻兩張的間隔不夠，會搞不清楚哪張是剛翻的，要調 delay
- AI 生成的 SCSS 結構很扁平，我重新用 BEM 命名法組織過一遍
- AI 生成的音效太刺耳，要自己調頻率、音量跟衰減曲線

所以 Vibe Coding 的正確心態應該是：**讓 AI 幫你跑完前 80%，你專心打磨剩下的 20%**。那 20% 往往才是使用者真正感受到的東西。

## 下一篇

到目前為止有四款趣味遊戲了，但我覺得還少了一塊——學習性質的內容。小孩光玩遊戲是很開心，但如果能順便認識數字、學學英文字母，這個樂園就更有價值了。

下一篇來講學習遊戲的設計，還有積分系統、學習履歷這些比較偏「產品面」的功能是怎麼做出來的。
