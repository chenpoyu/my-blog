## 「我們預期同時 2000 人在線」

三月底的需求會議上,客戶的產品經理說了這句話。

「我們預期上線後,尖峰時段會有 2000 個使用者同時在線。系統要能撐住。」

聽起來很簡單,對吧？

但我心裡想的是：「2000 個同時在線,到底是什麼意思?」

- 2000 個人同時登入?
- 2000 個人同時瀏覽?
- 2000 個人同時搶優惠券?

這些情境對系統的壓力完全不同。

「你們的使用情境是什麼?」我問。

「嗯⋯⋯大概就是會員平常使用,然後每週會有一次搶券活動。」

「搶券的時候有多少人?」

「可能 500-1000 人同時搶。」

好,這樣就清楚多了。

我們需要驗證兩個情境:
1. **日常使用**: 2000 人同時在線,但分散在各個功能
2. **搶券尖峰**: 500-1000 人同時執行複雜流程

## 不做壓測會怎樣?

有人問:「一定要做壓測嗎?我們開發時不是都測過了?」

是,我們有做功能測試、整合測試。但那些都是「低流量」的測試。

壓力測試要驗證的是:
- **系統在高流量時會不會掛掉**
- **回應時間會不會變得很慢**
- **哪裡是瓶頸** (CPU? 記憶體? 資料庫? 網路?)
- **系統的極限在哪裡**

如果不做壓測,上線後才發現問題,就慘了:
- 使用者體驗很差 (頁面跑很久)
- 系統掛掉 (直接 500 error)
- 客戶不爽 (「你們怎麼測的?」)
- 半夜被叫起來修 bug (最慘)

## 壓力測試的種類

我上網查了一下,壓力測試有很多種:

### 1. Baseline Testing (基準測試)
- 目的: 建立效能基準線
- 流量: 正常使用量
- 驗證: 系統在正常情況下的表現

這是最基本的,要先知道「正常情況」是什麼樣子。

### 2. Load Testing (負載測試)
- 目的: 驗證系統在預期負載下的表現
- 流量: 目標使用量 (例如 2000 併發)
- 驗證: 能否滿足效能需求

這是我們最需要的,要確保系統能撐住 2000 人。

### 3. Stress Testing (壓力測試)
- 目的: 找出系統的極限
- 流量: 超過預期的負載,持續增加直到系統崩潰
- 驗證: 系統在極限時的行為 (是 graceful degradation 還是直接掛掉?)

這個可以讓我們知道「安全邊界」在哪裡。

### 4. Spike Testing (尖峰測試)
- 目的: 驗證系統應對突然流量暴增的能力
- 流量: 突然從 100 併發暴增到 2000
- 驗證: 系統會不會因為突然的流量而掛掉

這個很重要,因為搶券活動就是這種情境。

### 5. Soak Testing (浸泡測試)
- 目的: 驗證系統長時間運行的穩定性
- 流量: 持續的中等負載,跑 24-48 小時
- 驗證: 是否有記憶體洩漏、連線洩漏等問題

這個可以找出一些「慢性病」,例如記憶體沒有正確釋放。

## 我們的測試計畫

根據客戶的需求,我們決定做這些測試:

| 測試類型 | 目標 | 預計時程 |
|---------|------|---------|
| Baseline | 建立基準 (100 併發) | 第 1 週 |
| Load | 驗證 2000 併發 | 第 2-3 週 |
| Spike | 驗證搶券情境 (500→1000 突增) | 第 3 週 |
| Stress | 找出系統極限 (2000→4000) | 第 4 週 |
| Soak | 24 小時穩定性測試 | 第 4 週 |

總共要 4 週。

「這麼久?」客戶有點驚訝。

「測試不是跑一次就好,」我解釋,「每次測試可能會發現問題,修完要再測一次。而且要測多種情境。」

「好吧,那就照你的計畫來。」

## 選什麼工具?

接下來要決定用什麼工具做壓測。

常見的選擇有:
- **JMeter**: 老牌工具,功能完整,但介面有點老舊
- **Gatling**: 用 Scala 寫的,效能不錯,但要學 Scala
- **Locust**: 用 Python,語法簡單,而且可以整合 Azure Load Testing
- **k6**: 用 JavaScript,輕量級,但不能整合 Azure
- **Azure Load Testing**: Azure 自家的服務,可以直接整合 App Service

我們最後選了 **Locust**。

原因:
1. **可以整合 Azure Load Testing**,這是最關鍵的
2. **語法簡單**,團隊的 Python 能力足夠
3. **可以用 CLI 執行**,方便整合 CI/CD
4. **免費且開源**

本來我想用 k6 的,以前專案都用它。但這次發現 **k6 不能整合 Azure Load Testing**,只能自己架 agent 機器。

既然都在 Azure 了,當然要用 Azure Load Testing,不用自己管機器。

## 下一步

這週先把 k6 裝起來,寫幾個簡單的測試腳本。

然後下週開始跑 Baseline Testing。

但在那之前,我還要先確認一件事:

**測試環境要跟正式環境一樣的規格嗎?**

如果測試環境用 S2,正式環境用 P2,那測試結果根本沒意義。

這又是一個要跟客戶討論的預算問題了⋯⋯

壓測不只是技術問題,還是成本問題、溝通問題。

這個專案真的是什麼都要自己來。
