除了佔 60% 的駐點大組，剩下的 40% 人力是以專案為單位進行編制。目前有四個專案組同時在跑，每個組 3-5 人，這種編制方式帶來了全新的管理挑戰。

之前當組長時，我只要管一個小組，大家做同一個專案，很單純。現在要同時管理多個專案組，複雜度完全不是線性成長，而是指數級的。

## 目前的專案組結構

**專案組 A：國防部專案**
- 人數：5人（1個 PM + 4個工程師）
- 技術棧：Java + Spring Boot + Oracle
- 特性：需求明確但變更流程繁瑣

**專案組 B：高公局專案**
- 人數：4人（1個 PM + 3個工程師）
- 技術棧：Java + Spring MVC + PostgreSQL
- 特性：需要高穩定性，不能出錯

**專案組 C：民間專案 A**
- 人數：3人（全工程師，我兼 PM）
- 技術棧：Go + PostgreSQL
- 特性：新技術嘗試，時程較彈性

**專案組 D：民間專案 B**
- 人數：3人（全工程師，外部 PM）
- 技術棧：Ruby on Rails + MySQL
- 特性：需求變動快，客戶很急

## 管理上的挑戰

### 挑戰 1：資源調度

最常遇到的情況是：某個專案很趕，某個專案較閒。

上個月就發生過：國防部專案突然要趕一個緊急需求，需要在兩週內上線。但我們組裡的人都滿載了，怎麼辦？

我看了看其他組，民間專案 A 剛好在測試階段，工程師比較不忙。我想：「把民間專案 A 的小王調過去支援兩週，應該可以吧？」

結果問題來了：

**小王的反應：**「我對國防專案不熟，要花時間了解，而且兩週後我還要回來繼續測試，來回切換很傷。」

**專案 A PM的反應：**「測試階段還是需要人，有 bug 要改，小王走了誰來處理？」

**國防專案 PM 的反應：**「兩週能上手嗎？我寧願自己的人加班。」

最後我協調了半天，決定：
- 小王支援一週（不是兩週）
- 只負責相對獨立的模組
- 民間專案 A 的測試由另一位工程師cover
- 國防專案的 PM 要準備好完整的文件和環境

這一來一回，光協調就花了兩天。

### 挑戰 2：技能差異

每個專案的技術棧都不太一樣：
- 國防和高公局用傳統的 Spring MVC
- 民間專案 A 用 Go
- 民間專案 B 用 Ruby on Rails

這導致人力很難彈性調度。

用 Java 的工程師調去支援 Go 專案？他得先學 Go。  
用 Go 的工程師調去 Java 專案？語法倒是會，但框架不熟。

有次我天真地想：「大家都是工程師，語言不就是工具嗎？互相學學就好。」

結果小張跟我說：「Poyu，我從 Java 轉 Go，光熟悉 goroutine 和 channel 就花了一週，更別說各種套件和最佳實踐了。」

我才明白：技術棧的差異是真實存在的成本。

### 挑戰 3：進度追蹤

同時管四個專案，要怎麼知道每個專案的狀況？

一開始我是每個專案都參加每日站會，結果發現：每天光開會就要 2 小時，自己的事都做不完。

後來我改成：
- 只參加每週的專案回顧會議
- 每日站會讓各組自己開，PM 整理重點給我
- 重要決策或卡住的問題才拉我進來討論

但這樣又有新問題：有時候問題已經拖了好幾天，我才知道。等我介入時，已經影響進度了。

**我需要找到一個平衡點：既不要微管理，也不能太放任。**

## 建立「人力池」概念

為了解決資源調度的問題，我建立了「人力池」的概念。

### 基本原則：

1. **核心成員固定** - 每個專案至少有 2 位核心成員，不輕易調動
2. **彈性成員可調** - 每個專案有 1-2 位彈性成員，可以跨組支援
3. **支援有時限** - 跨組支援最多兩週，避免長期分散精力
4. **技能矩陣** - 建立一個表格，記錄每個人會什麼技術

**技能矩陣範例：**

| 姓名 | Java | Go | Ruby | DB | 前端 | 可調度 |
|------|------|----|----|-----|------|--------|
| 小王 | ★★★ | ★★ | - | ★★ | ★ | Yes |
| 小李 | ★★★ | - | - | ★★★ | - | No (核心) |
| 小張 | ★★ | ★★★ | - | ★★ | ★★ | Yes |

這樣一來，當需要調人時，我可以快速找到：
- 誰有相關技能
- 誰目前可以調度
- 調過去的學習成本多高

### 實際運作

上週國防專案又需要支援，這次我用矩陣一看：

- 小王：Java ★★★，可調度 ✓
- 小張：Java ★★，目前在 Go 專案，進度ok

我選了小王，因為他 Java 能力強，上手快。調過去三天就開始產出了。

兩週後小王回來，雖然有點不適應，但總比之前好多了。

## 跨組協作機制

除了臨時支援，我還建立了一些跨組協作機制：

### 1. 技術週會

每週五下午，各組輪流分享：
- 遇到什麼技術問題
- 如何解決的
- 有什麼可以分享給其他組

這樣可以避免重複踩坑，也能促進跨組交流。

### 2. Code Review 跨組

偶爾讓其他組的人來 Review，好處是：
- 新視角，可能發現不同問題
- 讓大家了解其他專案在做什麼
- 學習不同的代碼風格

### 3. 共用元件庫

我們建立了一個共用的元件庫，把各專案常用的功能抽出來：
- 日誌模組
- 錯誤處理
- 資料驗證
- 工具函式

這樣可以減少重複開發，也讓新人能更快上手。

## 平衡的藝術

管理多專案最困難的是「平衡」。

**不要讓工程師頻繁切換專案**  
context switch 的成本很高，一個人如果一週切三個專案，效率會非常低。

我現在的原則是：非必要不調人，調了就至少兩週，讓他能專注做完一個階段。

**不要讓某個專案長期當救火隊**  
如果某個專案老是借不到人，或老是被抽走人，PM 會很不爽，團隊士氣也會受影響。

我會記錄每次的人力調動，確保公平性。

**不要為了調度而調度**  
有時候某個專案閒一點也沒關係，不用強制讓每個人都滿載。人需要喘息的空間。

## 目前的成效

實施這套機制三個月後：

**正面效果：**
- 跨組支援次數從每月 8 次降到 3 次（說明資源規劃更準確）
- 支援平均時長從 3 週降到 1.5 週
- 專案 PM 滿意度提升（抱怨變少了）

**還在優化的：**
- 技能矩陣需要持續更新（有人學了新東西要記錄）
- 彈性成員的工作安排（不能讓他們覺得自己是救火隊）
- 長期的人力規劃（如何預測未來的需求）

## 反思

以專案為單位編制團隊，確實能提升專案的獨立性和效率，但也帶來協調的複雜度。

**關鍵是要建立明確的機制：**
- 誰可以調？什麼情況下調？調多久？
- 如何公平分配資源？
- 如何確保知識在組與組之間流動？

**更重要的是溝通：**
- 跟 PM 溝通：為什麼要調人，調多久
- 跟工程師溝通：你去支援的價值是什麼，不是懲罰
- 跟團隊溝通：我們是一個大團隊，互相幫助是應該的

管理永遠沒有完美的模式，只有持續優化的過程。
