面試了這麼多人之後，我發現面試題目的設計其實是一門學問。題目不是越難越好，而是要能在有限的時間內，看出候選人的真實能力。

最近在檢討面試流程時，我發現有些題目根本看不出候選人的實力，有些題目又太簡單，區分度不夠。於是我重新設計了一套面試題庫。

## 我常用的題目類型

### 1. 基礎觀念題

這類題目用來確認候選人的基本功是否扎實。

**範例：**
- 「說明 Java 的多型是什麼？」
- 「HTTP 和 HTTPS 的差異是什麼？」
- 「什麼是 SQL Injection，如何預防？」

這些題目看起來簡單，但能快速篩選出完全不適合的候選人。如果連基本觀念都說不清楚，後面就不用浪費時間了。

但我不會只問定義，還會追問：「你在實際專案中怎麼應用這個概念？」這樣才能分辨是真的懂，還是只是背答案。

### 2. 情境題

這類題目最能看出候選人解決問題的思路。

**範例：**
- 「系統突然變慢，你會怎麼排查？」
- 「如果要設計一個檔案上傳功能，你會考慮哪些面向？」
- 「當兩個 API 回傳的資料不一致，你會如何處理？」

我最看重的是候選人的思考過程。他會不會先問清楚情境？會不會有系統化的排查方法？還是只會亂猜？

上週有個候選人在回答「系統變慢怎麼辦」時，先問我：「變慢是指回應時間變長，還是吞吐量下降？是突發性的還是持續性的？」

這種能先釐清問題再解決的人，通常都不錯。

### 3. 開放式問題

這類問題用來了解候選人的經驗深度和思考廣度。

**範例：**
- 「你做過最複雜的專案是什麼？遇到什麼困難？」
- 「如果讓你重新設計之前的系統，你會改什麼？」
- 「你最近學了什麼新技術？為什麼選它？」

這種問題沒有標準答案，但能看出很多東西：

有個資深候選人跟我說，他之前做的系統如果重來，會把單體架構改成微服務。我問他為什麼，他說：「因為當時團隊擴大後，大家改同一個 codebase 常常衝突，部署也很慢。但微服務也有代價，像是分散式事務和監控會變複雜...」

這種能看到 trade-off 的人，就是我要的。

### 4. 實作題

實作題是最直接的評估方式，能看出候選人的程式設計能力。

**我的原則：**
- 題目要有代表性，不要太偏門
- 難度適中，15-20分鐘能完成
- 可以用任何語言，不限定
- 重點是思路，不是語法細節

**常用題目：**
```
- 反轉字串或陣列
- 判斷迴文
- 找出陣列中重複的元素
- 實作簡單的資料結構（Stack, Queue）
```

我會在旁邊觀察他怎麼寫：
- 會不會先問清楚需求
- 會不會考慮 edge case
- 遇到卡住的時候怎麼反應
- 寫完會不會自己測試

有次一個候選人寫完後，主動說：「這個解法時間複雜度是 O(n²)，如果資料量大的話，可以用 HashMap 優化成 O(n)。」這種主動思考優化的態度，加分。

## 重點不是答案，而是過程

我面試時最常說的一句話是：「慢慢想，說出你的思路。」

有些人答錯了，但思路清晰，我會引導他找到正確方向；有些人答對了，但問深一點就露餡了，很明顯是死背。

**面試就是要分辨這些差異。**

上個月有個候選人，我問他「如何設計一個快取機制」，他回答得很標準：LRU、TTL、快取穿透...全都答出來了。

但我追問：「你實際專案中用過嗎？」他說沒有，只是看文章學的。

另一個候選人，雖然答得沒那麼完整，但他說：「我們專案中用 Redis 做快取，一開始沒設 TTL，結果記憶體爆了。後來設了過期時間，又遇到快取同時失效導致資料庫壓力突然暴增...」

很明顯，後者的回答更有價值。

## 面試題庫的維護

我現在維護了一份面試題庫，分成：
- 資淺工程師題目
- 資深工程師題目
- 技術主管題目

每次面試完，我會記錄：
- 哪些題目效果好
- 哪些題目太簡單或太難
- 候選人的回答模式

這樣持續調整，才能讓面試越來越精準。

## 結論

設計面試題目的目標很簡單：**在有限時間內，盡可能準確地評估候選人。**

- 不要為了難而難
- 要能區分不同程度的候選人
- 要能看出思考過程，不只是答案
- 要符合實際工作需求

面試是雙向的，好的題目不只能評估候選人，也能展現公司的專業度。
