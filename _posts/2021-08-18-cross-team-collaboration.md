「Poyu，國防專案那邊很趕，可以支援一個人嗎？」

這句話我幾乎每週都會聽到。當 A 專案很趕，需要從 B 專案借人時，聽起來很簡單，但實際執行起來卻是一門藝術。

上週就發生了一次典型的跨組協調案例，讓我深刻體會到這件事的複雜度。

## 一次失敗的借調

**情境：**  
國防專案的 PM 小林週一早上衝進我辦公室：「Poyu！客戶突然要加一個緊急功能，要在週五demo，我們人手不夠！」

我問：「需要幾個人？」  
「至少兩個，最好是會 Java 的。」

我看了看其他專案的狀況，民間專案 A 的小王和小陳這週比較不忙，我想：「那就他們兩個吧。」

我打電話給小王：「小王，國防專案那邊很急，你這週去支援一下。」

小王沉默了幾秒，說：「可是...我手上還有測試要做，而且我對國防專案完全不熟。」

「沒關係，他們會教你，測試可以晚一週。」

小王不太情願地答應了。

**結果：**
- 週一下午，小王和小陳到國防專案報到，花了半天在設定環境
- 週二，他們開始看代碼，發現架構跟他們的專案完全不同
- 週三，他們才開始寫程式，但頻頻卡住，一直在問問題
- 週四，小林跟我抱怨：「這兩個人問題太多，我的人光解答就耗掉很多時間」
- 週五，功能勉強完成，但品質不如預期，demo 時出了幾個小 bug

**而且：**
- 小王和小陳抱怨：「被臨時抓去支援，打亂自己的節奏」
- 民間專案 A 的進度延遲了
- 國防專案雖然勉強完成，但多花了很多溝通成本

這次借調是失敗的。

## 常見的問題

經過這次教訓，我整理出跨組支援常見的問題：

### 1. 被借調的人會抱怨

「我自己的事都做不完，為什麼還要去支援別人？」  
「我不熟那個專案，去了也幫不上忙。」  
「每次都叫我，為什麼不叫別人？」

這些抱怨都很合理。工程師都有自己的工作和節奏，被打斷確實不舒服。

### 2. 原專案 PM 不願意放人

「我的專案也有時程壓力，人被抽走怎麼辦？」  
「他是我的核心成員，走了誰來頂？」  
「為什麼總是我們這組被借人？」

PM 都希望保護自己的團隊和進度，不願意輕易放人。

### 3. 借調後的績效如何計算

工程師去支援了兩週，這兩週的成果算哪個專案的？  
考核時怎麼評估？  
兩邊都做了，會不會兩邊都覺得貢獻不夠？

### 4. 上手成本被低估

很多人以為：「都是寫程式，換個專案應該很快就能上手吧？」

實際上：
- 要熟悉新的代碼架構
- 要了解業務邏輯
- 要適應不同的開發流程
- 可能還要學新的技術或工具

這些都需要時間。

## 改善後的做法

經過幾次失敗，我建立了一套跨組支援的機制：

### 1. 事先溝通，不是臨時抓人

現在每週一的主管會議，我會詢問：
- 各專案未來兩週的人力需求
- 預期會有哪些緊急狀況
- 哪些專案有餘裕可以支援

這樣可以提前規劃，而不是臨時抱佛腳。

**範例：**  
上週會議，小林說：「我們下下週可能需要一個人支援。」  
我立刻協調：「小王，你下週任務結束後，可以準備一下國防專案的環境和文件，下下週去支援。」

這樣小王有一週的準備時間，支援時就順利多了。

### 2. 明確時間和範圍

我現在借調都會明確：
- 支援多久（通常 1-2 週，不超過一個月）
- 負責什麼範圍（只做某個模組，不是整個專案）
- 何時開始、何時結束
- 書面記錄（發郵件給相關人員）

**範例：**
```
主旨：人力支援通知 - 小王支援國防專案

支援人員：小王
支援專案：國防專案
支援時間：8/23 - 9/3 (兩週)
支援內容：負責報表模組的開發
原專案：民間專案 A（進度由小陳 cover）
```

這樣大家都清楚，不會有模糊地帶。

### 3. 雙向加分的績效機制

我跟 HR 協調，建立了「跨組支援加分」機制：

**被支援專案：**
- 完成的功能算該專案的成果

**原專案：**
- 這段時間的進度延遲不算延誤（有正當理由）

**被借調的工程師：**
- 兩邊的績效都計算
- 額外加「團隊協作」加分
- 跨專案經驗對職涯發展有幫助

這樣一來，被借調不是懲罰，而是機會。

### 4. 技能成長的角度

我會跟被借調的工程師說：

「這次去支援，你可以學到國防專案的架構設計，他們處理高安全性需求的經驗對你很有幫助。」

「你去幫忙 Ruby 專案，可以接觸新的技術棧，對職涯發展有好處。」

**讓支援變成學習機會，而不是負擔。**

有位工程師後來跟我說：「之前覺得被派去支援很煩，但確實學到不少東西，而且認識了其他組的同事。」

### 5. 建立支援 SOP

我們現在有一套標準流程：

**支援前：**
1. 借調確認：需要什麼技能、支援多久
2. 人選評估：誰最適合、誰有餘裕
3. 雙向溝通：跟原 PM 和被借調者溝通
4. 準備資料：提供專案文件、環境說明

**支援中：**
1. 第一天：環境設定、專案導覽
2. 第二天：閱讀代碼、了解業務
3. 第三天開始：實際任務，但有專人可諮詢
4. 每日回報：進度如何、有什麼困難

**支援後：**
1. 完成交接：文件更新、知識分享
2. 回饋收集：這次支援學到什麼、有什麼建議
3. 績效記錄：兩邊都要記錄貢獻

有了SOP，每次借調就不會手忙腳亂。

## 公平性很重要

跨組支援最怕的就是不公平。

如果總是某幾個人被借調，他們會覺得被針對。  
如果總是某個組被借人，那個組會覺得被剝削。

**我會記錄每次的借調：**

| 被借調者 | 原專案 | 支援專案 | 時間 | 頻率 |
|---------|--------|---------|------|------|
| 小王 | 民間A | 國防 | 2週 | 今年第2次 |
| 小李 | 高公局 | 民間B | 1週 | 今年第1次 |
| 小陳 | 民間A | 國防 | 2週 | 今年第3次 |

看到小陳已經第3次了，下次就不找他，找其他人。

**互相幫忙是團隊文化的一部分，但前提是公平。**

## 什麼情況不適合借調

不是所有情況都適合跨組支援：

**不適合的情況：**
1. **技術差異太大** - Java 工程師去支援前端？學習成本太高
2. **時間太短** - 只需要幾天，光上手就要花掉大半時間
3. **關鍵時刻** - 兩個專案都在關鍵期，誰都不能缺人
4. **人格不合** - 有些人就是處不來，強制合作會更糟

遇到這些情況，寧願想其他辦法：
- 加班趕工
- 調整時程
- 砍需求
- 外包支援

## 現在的效果

建立這套機制後，跨組支援順暢多了：

**工程師的反饋：**
- 「至少知道要支援多久，不會無限期拖著」
- 「有學到東西，不是白做」
- 「績效有算到，不吃虧」

**PM 的反饋：**
- 「提前知道人力會被借走，可以先規劃」
- 「支援的人有準備，上手快多了」
- 「借調機制很透明，比較能接受」

**數據改善：**
- 借調失敗率從 40% 降到 10%
- 平均上手時間從 3 天降到 1.5 天
- 因借調產生的抱怨減少 70%

## 結論

跨組支援是必要的，但要做得好需要：

1. **提前規劃** - 不要臨時抱佛腳
2. **明確規則** - 時間、範圍、績效都要清楚
3. **雙贏思維** - 讓所有人都能獲益
4. **公平對待** - 不要總是找同樣的人
5. **持續優化** - 根據反饋調整機制

**互相幫忙是團隊文化的一部分，但前提是要有明確的規則和公平性。**

沒有規則的互助，最後會變成剝削；  
有了規則的協作，才能長期運作。
