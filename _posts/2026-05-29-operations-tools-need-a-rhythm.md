## 工具不是清單，是排查路徑

前兩篇先談了兩件事。

第一，維運要先讓問題被看見。  
第二，問題要先被翻譯成需求、流程和責任分工。

到這裡才適合談工具。

因為工具如果沒有接到前面的脈絡，
很容易變成一份看起來很完整，
但真正排查時還是很分散的清單。

維運中心裡一整排服務跑起來時，
我看著這些名字，
其實第一個想法不是「工具終於齊了」。

而是：

「它們能不能串成一條路？」

Prometheus。  
Grafana。  
Alertmanager。  
Elasticsearch。  
Kibana。  
Logstash。  
Filebeat。  
Vault。  
Dockge。  
SonarQube。  
Trivy。  
GitHub Actions Runner。

工具很多。

但工具多，不代表維運成熟。

有時候剛好相反。

如果每個工具都只是單獨存在，
大家要查問題時還是各開各的頁面，
那只是把資訊分散到更多地方。

真正有價值的不是工具清單。

而是這些工具能不能形成一個排查和決策的節奏。

## 指標工具，回答的是現在健不健康

Prometheus、Grafana、node_exporter、cAdvisor、Blackbox Exporter，
我會把它們放在同一類。

它們主要回答的是：

- 服務有沒有活著
- 回應時間有沒有變慢
- CPU、記憶體、磁碟是不是快不夠
- Container 有沒有重啟或 OOM
- 外部端點探測有沒有失敗

這些是維運的儀表。

沒有它們，
團隊常常只能等使用者回報才知道問題。

但只有它們也不夠。

因為指標通常只能告訴你：

「這裡不太對。」

它不一定會直接告訴你為什麼。

## 日誌和錯誤工具，回答的是發生了什麼

所以第二層會接到 log 和 error。

Filebeat 把容器輸出的 log 收進 Elasticsearch。  
Logstash 處理比較主動送進來的應用日誌。  
Kibana 讓人能搜尋、過濾、比對時間點。  
GlitchTip 或 Sentry-compatible webhook 則比較適合收 application exception。

這一層要解的是：

- 哪支服務吐錯
- 錯誤發生在哪個版本
- traceId 或 requestId 能不能串起來
- 是單一使用者還是很多使用者都遇到
- deployment 之後錯誤量有沒有突然上升

指標看到煙。

日誌和錯誤工具讓你靠近火源。

但這裡也有一個前提：

應用程式自己要願意好好說話。

如果 log 都是零散字串，
沒有 service、level、traceId、userId、release，
那就算 ELK 架得再漂亮，
排查還是會痛。

## 告警工具，回答的是誰該被叫起來

Alertmanager 這一類工具，
我覺得最容易被誤會。

很多人以為告警就是把錯誤推到群組。

但好的告警不是吵。

好的告警應該要有分級、去重、靜默和路由。

它要知道：

- 哪些告警真的需要立刻處理
- 哪些只是觀察
- 哪些可以上班時間處理
- 同一個問題不要一直重複轟炸
- 維護期間可以暫時 silence
- 告警要送到對的人或工作流

如果告警沒有設計，
最後團隊會被通知淹沒。

久了之後，
大家就會開始忽略告警。

這比沒有告警還危險。

## 部署和品質工具，回答的是這次變更能不能進來

維運不能只從 production 端開始看。

很多事故其實在進 production 之前，
就已經有跡象了。

所以我會把 GitHub Actions Self-hosted Runner、SonarQube、Trivy 放進維運視角裡。

它們處理的是另一種問題：

- 測試有沒有過
- Quality Gate 有沒有守住
- 安全掃描有沒有高風險 CVE
- image 是不是從同一套流程 build 出來
- 部署成功或失敗有沒有被記錄

這些不是純開發流程。

它們其實是維運的前哨站。

因為越早把品質和風險擋下來，
後面 production 要承受的壓力就越少。

## 機密和參數工具，回答的是誰可以改什麼

Vault 和 Dockge 看起來跟監控比較不直接。

但實際上很重要。

一個系統出問題，
常常不是因為程式突然壞掉。

而是某個環境變數改了。  
某個 token 過期了。  
某個 OAuth secret 沒有同步。  
某個 stack 重啟後讀到舊設定。

如果密碼、token、feature flag、port、log level 全部混在 `.env` 和口頭交接裡，
維運一定會很難追。

所以我喜歡把它們分開：

- Vault 放敏感機密
- Dockge 管非敏感部署參數
- Git 管可以版控的設定
- Keycloak 管身份和角色

這樣不是為了架構漂亮。

而是出事時，
比較知道哪一類變更要去哪裡查。

## 工具要串成一條路

所以看維運工具時，
我比較少問「還缺哪個工具」。

我比較常問：

「如果明天某個服務出事，這些工具能不能串成一條路？」

理想上應該是這樣：

部署事件進 Deploy Timeline。  
錯誤事件進 Error Timeline。  
Prometheus 告警進 Alert Timeline。  
最後全部收進 Ops Timeline。

排查時不用在十個畫面之間靠記憶拼圖。

而是能先看到時間順序：

哪個版本剛部署。  
哪個服務先出錯。  
哪個告警後來 firing。  
哪個 Incident 還沒 resolved。

這樣工具才不是各自存在。

它們才真的變成維運系統的一部分。

## 最後還是回到人怎麼用

工具可以買，可以裝，可以用 Docker Compose 很快拉起來。

但工具不會自己變成流程。

它需要團隊決定：

- 哪些指標值得看
- 哪些告警真的要叫人
- 哪些 log 欄位是必填
- 哪些部署要被記錄
- 哪些 secret 不能再放在 repo
- 哪些事件要進 Incident 流程

這些判斷不做，
工具越多只是越忙。

這一篇想留下來的，
不是一份很長的工具清單。

而是一套查問題時大家會自然走的路。

從指標看到異常，
從日誌靠近原因，
從部署紀錄確認變更，
從告警判斷嚴重度，
從 Incident 留下處理結果。

工具真正有價值的地方，
是讓這條路越走越清楚。

系列最後一篇，
就要把這條路再往上收。

工具串起來之後，
還要變成服務清單、事件時間線、Incident 流程和營運指標，
才真的像一個可以支撐日常工作的營運系統。
