到了這週，我學習建立專案風險登錄冊。我才意識到風險管理是專案管理很重要的一環。

風險是不確定的事件，如果發生會對專案目標產生正面或負面影響。大部分時候我們談的是負面風險，也就是威脅。正面風險叫機會，也要管理。

風險管理有五個步驟。規劃風險管理、識別風險、分析風險、規劃風險回應、監控風險。這是一個持續的過程，不是一次性的活動。

識別風險時，要從各個角度思考。技術風險、資源風險、時程風險、成本風險、外部風險等。可以用腦力激盪、檢核表、過往專案經驗來識別。

我們團隊花了半天時間做風險識別。每個人寫下自己想到的風險，然後一起討論整理。總共列出了二十幾項風險，包括關鍵人員離職、第三方 API 不穩定、需求變更、時程太緊等。

識別風險後要分析。評估每個風險的發生機率和影響程度。通常用高中低三個等級，然後用矩陣圖呈現。高機率高影響的是首要關注的風險。

風險分析可以是定性的，也可以是定量的。定性分析比較快，用主觀判斷。定量分析比較精確，用數據計算，但需要更多時間和資料。我們先做定性分析，重大風險再做定量分析。

分析完後要規劃回應策略。對威脅有四種策略：規避、轉移、減輕、接受。規避是改變計畫讓風險不會發生，轉移是讓別人承擔風險，減輕是降低發生機率或影響，接受是不採取行動。

對機會也有四種策略：開拓、分享、提升、接受。基本上是對應威脅策略的正面版本。

我們對每個高風險項目都規劃了回應策略。比如關鍵人員可能離職，我們的策略是減輕，做法是知識文件化和交叉訓練。第三方 API 不穩定，策略是減輕，做法是建立備用方案。

每個風險還要指派負責人，負責監控風險和執行回應計畫。不能只是列在文件上，要有人真正追蹤。

建立了風險登錄冊，記錄所有識別的風險、分析結果、回應策略、負責人、狀態。這個文件要定期更新，在每週專案會議中檢視。

也建立了風險升級機制。如果風險狀況惡化或超過團隊能處理的範圍，要向上呈報給高層決策。

兩週後，真的有一個風險發生了。第三方 API 突然改版，導致我們的整合失效。因為有事前準備，我們啟動了備用方案，影響控制在一天內。如果沒有風險管理，這個問題可能會讓專案停擺一週。

這個經驗讓我理解風險管理的價值。不是說做了風險管理就不會出問題，而是出問題時有準備，可以快速應對。預防勝於治療。
