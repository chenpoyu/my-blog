面試這麼多人，大部分都在預期範圍內，但偶爾會遇到讓人眼睛一亮的候選人，那種感覺就像挖到寶一樣。

上週面試了一位只有兩年經驗的工程師，履歷看起來很普通，第一家公司待了一年多，第二家公司做了半年多。我本來想說應該就是一般的資淺工程師，結果一聊才發現完全不是這樣。

## 讓我驚艷的面試過程

一開始我照慣例問基礎題：「可以說明一下 Java 的記憶體管理嗎？」

他很快地回答：「JVM 的記憶體主要分成 Heap 和 Stack。Heap 是存放物件實例的地方，所有 new 出來的物件都在這裡；Stack 則是存放方法呼叫和區域變數，每個執行緒都有自己的 Stack...」

這些都是標準答案，我正準備問下一題，他卻主動延伸：

「Heap 又可以細分成 Young Generation 和 Old Generation。Young Gen 裡面有 Eden Space 和兩個 Survivor Space。新物件通常會先配置在 Eden，經過幾次 Minor GC 還存活的，就會被移到 Old Gen...」

他越講越深入，開始談 G1 GC 和 ZGC 的差異，還分享他在上一個專案中遇到 Full GC 導致停頓的問題，怎麼透過調整 `-Xmx` 和 `-XX:MaxGCPauseMillis` 參數來優化。

我心裡想：「這真的只有兩年經驗嗎？」

## 繼續深入測試

為了確認不是背答案，我問了一些比較實務的問題：

「你有用過 profiler 工具嗎？」  
「有，我用過 JProfiler 和 VisualVM。上次有個 API 回應很慢，我用 JProfiler 找到瓶頸在某個 SQL 查詢，後來加了索引就解決了。」

「多執行緒有實際寫過嗎？」  
「有，我做過一個批次處理程式，要處理幾萬筆資料。一開始用單執行緒跑很慢，後來改用 ExecutorService 配合 ThreadPoolExecutor，速度快了五倍。但要注意執行緒安全問題，像是我用 ConcurrentHashMap 來避免競爭條件...」

每個回答都很具體，而且能說出背後的原理和實際遇到的坑。

## 天份還是努力？

面試結束後，我很好奇問他：「你怎麼會對這些這麼熟？」

他笑著說：「我就是很喜歡研究技術。平常下班會看一些技術文章和開源專案的 source code，週末有時候會寫一些 side project 來試新東西。」

原來如此。這不只是天份，更多的是熱情和自主學習的能力。

他還說：「其實我在現在公司有點大材小用，都在做很重複的 CRUD，學不到什麼新東西，所以想換環境。」

這我完全理解。有些公司的專案確實比較單純，無法讓這種有潛力的人發揮。

## 如何用對人才

當下我就決定要錄取他，但也開始思考：

**如何讓這樣的人才在團隊中發揮最大價值？**  
不能讓他也去做簡單的 CRUD，那是浪費。應該讓他參與比較有挑戰性的專案，或是負責技術研究和導入的工作。

**如何避免大材小用？**  
要給他成長的空間和學習的機會。如果只是把他當普通工程師用，他可能很快又會想離職。

**如何平衡團隊公平性？**  
如果給他特殊待遇，其他資淺工程師會不會有意見？這需要溝通和管理。

## 後續

他加入團隊後，我把他放在一個公部門專案裡，負責比較複雜的模組。果然表現很好，而且很主動幫助其他組員解決技術問題。

三個月後，我讓他帶一位新人，他也教得很好。現在他已經是團隊裡的技術骨幹之一。

**選對人只是第一步，用對人才是關鍵。**

這次經驗讓我學到：有些人的潛力遠超過履歷上顯示的，面試的時候要多挖掘，不要只看表面。
